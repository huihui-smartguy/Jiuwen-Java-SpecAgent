## 一、背景与动机

- **已解决的问题（v1.0）**：`TestKnowledgeBase/Fault/` 与故障匹配链路让 AutoTestFlow 具备了"应测哪些失败模式"的缺陷经验，能补足 happy-path 过重、SSE/超时/并发/错误质量缺口。
- **仍存在的缺口**：故障库回答"什么容易坏"，但不回答"测试套件是否专业、AI eval 是否可信、上线门禁是否充分、报告是否可行动"。
- **新需求**：将 Google、Microsoft、AWS、Netflix、Meta、OpenAI、Anthropic、NIST、OWASP 等公开实践与标准沉淀为可复用知识，使 AutoTestFlow 未来能在 stage3b/stage5 生成测试组合建议、AI 评测门禁、发布就绪结论与专家级报告。
- **核心决策**：新增 `Professional_experience/` 作为第三类知识源：它不替代 `contract.md`，不直接升级断言级别；只提供测试策略、验收门禁、AI eval 标准和报告质量标准。强断言仍必须来自 `contract.md`。

## 二、总体架构（知识分层）

```text
需求 / 源码 / 运行样本
   │
   ├── contract.md
   │      └─ 当前 SUT 承诺什么：唯一强断言 Oracle
   │
   ├── TestKnowledgeBase/Fault/
   │      └─ 什么容易坏：故障导向用例、历史缺陷回归、fault_ref
   │
   └── TestKnowledgeBase/Professional_experience/
          └─ 什么叫测得专业且可上线：测试组合、AI eval、发布门禁、报告质量
```

三条贯穿原则：

1. **Contract-first**：专业经验只能影响"测什么、如何验收、何时人工确认"，不能凭经验文本生成 L2 强断言。
2. **文件即协议**：专业经验以 Markdown + JSON 文件沉淀，未来可由确定性脚本或 stage5 汇总器条件消费。
3. **故障库协同**：Fault 库负责失败模式覆盖，Professional_experience 库负责测试负责人判断力；二者在用例设计与报告阶段汇合。

## 三、三轮深度分析结论

### 第一轮 —— AutoTestFlow 仓库架构

AutoTestFlow 已形成"LLM 编排 + 文件协议 + 契约约束 + 五分类结果"的稳定内核：

| 架构不变量 | 对 v4.0 的启发 |
|---|---|
| `contract.md` 是唯一强断言 Oracle | 专业经验不能越权成为判据 |
| `.state/*`、`test_design.json`、`case_results.json` 均为文件协议 | 新经验必须文件化、结构化 |
| 五分类结果区分 pass / harness_defect / sut_unsatisfied / sdk_defect / env_issue | 专业验收应帮助报告区分缺陷、风险、观察和环境问题 |
| 规格库就位后人工门可逐步降级 | 专家经验需要可积累、可追溯、可机器读取 |

结论：仅靠需求和契约能保证"断言可信"，但不能保证"测试策略成熟"。

### 第二轮 —— 故障库能力边界

故障库已经覆盖 Web、REST API、Agent、DFX 四类故障，通过 `trigger_pattern`、`validation_points`、`severity`、`tags` 驱动异常/边界/可靠性用例。

但故障库的自然边界是"缺陷模式"，它不负责：

- 测试金字塔和反馈速度是否健康；
- E2E 是否过多、是否低可定位；
- AI eval 数据集、grader、红队、安全和生产漂移是否完备；
- stage5 报告是否达到工程团队可采纳的 release readiness 质量。

结论：故障库是"缺陷经验记忆"，仍需要专业经验库承载"测试负责人判断力"。

### 第三轮 —— 全球工程实践

本轮归纳出多家公司和公开标准的一致共识：

| 共识 | 来源示例 | AutoTestFlow 启发 |
|---|---|---|
| 快速、稳定、可定位的反馈优先于堆 E2E | Google 测试金字塔 / SWE testing | stage3b 应关注测试组合健康度 |
| 测试是变更门禁，失败应阻断合并或发布 | Microsoft Engineering Fundamentals | stage5 应输出可执行门禁 |
| 可靠性需要故障演练、canary、恢复和监控证明 | Google SRE / AWS Well-Architected / Netflix FIT & Kayenta | DFX 与发布就绪要成为显式验收项 |
| 自动化测试报告必须低误报、可复现、可行动 | Meta Sapienz | trace、oracle_refs、fault_ref 必须贯穿报告 |
| AI 系统需要 eval dataset、grader、红队和生产监控 | OpenAI Evals / Anthropic eval guidance / Microsoft AI Observability | Agent/LLM 测试必须超越答案对错 |
| AI 安全有特有威胁模型 | NIST AI RMF / OWASP LLM Top 10 | prompt injection、过度自治、敏感信息、工具边界必须进入验收 |

结论：AutoTestFlow 要成为测试 agent，必须同时拥有"故障覆盖能力"和"专家验收能力"。

| `.state/professional_acceptance.json`（建议产物） | `autotestflow_mapping.md` | 未来 stage5 可生成的专业验收门禁结果 |
| `.state/ai_eval_readiness.json`（建议产物） | `autotestflow_mapping.md` | 未来 AI/Agent 测试就绪度结果 |

## 六、验收标准覆盖范围

`acceptance_criteria.json` 共 14 条：

| 类别 | 标准 |
|---|---|
| 软件测试 | 快速反馈层、E2E 比例控制、契约覆盖、故障模式覆盖、可观测性 |
| 可靠性 / 发布 | DFX 可靠性、Canary 和渐进发布就绪 |
| AI 测试 | AI 成功标准、Eval 数据集、评分器可靠性 |
| Agent 测试 | 工具调用验收 |
| AI 安全 / 运营 | AI 安全与红队、生产监控和漂移 |
| 报告质量 | 测试报告可行动性 |

每条标准均包含：

- `source_refs`：可追溯来源；
- `intent`：验收意图；
- `acceptance_checks`：可执行检查项；
- `evidence_required`：需要哪些文件、trace、指标或配置证据；
- `release_gate`：PR、release、AI release、production rollout 等门禁建议。

## 七、行为与兼容性

- **稳定流水线零行为变化**：本轮只新增知识文件，不改 `AutoTestFlow/SKILL.md`、稳定 `scripts/`、稳定 `templates/`、`reference/` 或 `shared/`。
- **不越权断言**：所有文档都明确 `Professional_experience/` 只提供策略和门禁，不替代 `contract.md`。
- **可渐进接入**：未来可在 stage3b 添加 `acceptance_refs`，在 stage5 添加 `professional acceptance` / `AI evaluation readiness` 矩阵；未接入时完全无影响。
- **可审计来源**：所有公司实践与验收标准均通过 `source_refs` 指向 `source_registry.json`，便于评审和更新。
