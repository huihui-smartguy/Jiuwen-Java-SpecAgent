# v5.0 Stage6 迭代2：Fault Analysis 与 Evidence Issue

## 背景

原 Stage6 仅围绕 `sdk_defect` 生成自动修复补丁，无法充分承接 `TestKnowledgeBase/Fault` 中 Web、REST API、Agent、DFX 以及未来更多故障模式的诊断需求。本次迭代将 Stage6 从“SDK 修复通道”升级为可扩展的 `fault_analysis` 分析层。

## 主要变更

- 新增 `.state/fault_analysis/analysis_plan.json` 与 `.state/fault_analysis/manifest.json`，用于承载 patchable、issue-only、diagnostic 等多类分析目标。
- 新增 `shared/fault_analysis_profiles.json`，以数据化 profile 支持 Web / REST / Agent / DFX / Security / Observability，并为未来故障模式提供默认 profile。
- 保留 `.state/remediation/plan.json` 作为 Stage7 本地应用 / 复验的兼容子集，仅包含 spec-required 且可安全 patch 的 contract defect。
- 新增 `templates/stage6_fault_analyze.md`，通用处理非 patchable 或未来故障模式，输出 `evidence.json`、`root_cause.md`、`fix_solution.md`、`issue.md`、`confidence.json`。
- 更新 `templates/stage6_defect_analyze.md` 与 beta 模板，patchable 目标也必须输出 evidence / fix_solution，并移除 PR 叙事。
- Stage7 移除自动 PR 提交，不再 push fork，不再创建 PR；仅在人工门 + `allow_open_issue=true` 后提交 evidence issue。
- `submit_remediation.py` 生成 issue body 时合并 Stage6 issue、修复方案、patch 摘要、回归测试摘要与 Stage7 `reverify.json` 实证结果。
- 配置 schema 升级到 v2.1，废弃 `fork_url`、`fork_remote_name`、`pr.*`、`allow_push`、`allow_open_pr`；新增 `require_evidence_before_issue`，兼容旧 `require_green_before_pr`。

## 安全纪律

- `contract.md` 仍是唯一 oracle；故障库与 wiki 只作 advisory。
- 只有 `class=sdk_defect` 且 contract authority 为 `spec-required` 的目标可进入 patch/reverify。
- 非 patchable 目标只生成 evidence issue 材料，不强行改码。
- Stage6 不做任何外发副作用；Stage7 只提交 evidence issue。

## 验证

- 新增 `AutoTestFlow/tests/test_stage6_iteration2.py`，覆盖 patchable 子集、DFX/未来 profile、配置兼容与 issue-only submitter。
- 更新 `examples/a2a/remediation_demo`，离线验证 plan → manifest → apply/reverify → dry-run issue 提交流程。
