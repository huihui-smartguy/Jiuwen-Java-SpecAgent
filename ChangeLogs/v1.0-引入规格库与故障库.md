## 一、背景与动机

- **问题**：以 `examples/a2a` 为锚点测 spring-ai-ascend 时，生成用例约 58% 是 happy-path，仅 2 条真实负向，流式/超时/并发/资源/错误质量 ≈ 0 覆盖——"以验证场景通畅为前提，**测不出模块/缺陷**"。
- **缺口**：流水线判据（Oracle）来自"需求 + 真实契约"，能保证"形态正确、流程跑通"，但**缺少"应测哪些失败模式"的缺陷经验**。
- **载体**：用户在 `Specification_Repository/` 沉淀了故障库（`rest_api_common_faults.json`：8 类 50 通用故障 + 5 历史缺陷）。
- **核心决策**：把故障库做成**可插拔的外部知识源（规格库）经文件注入**，**不硬编码**进 Skill 提示词——契合流水线"文件即协议 / 零知识调度"内核，且正是 `DESIGN.md`/`SKILL.md`/`spec.md` 早已预留的"规格库 seam"。（七维对比与论证见 `FaultsAnalysis/03` §1。）

## 二、总体架构（数据流）

```
故障库 Specification_Repository/rest_api_common_faults.json  [+ project_faults.json overlay]
        │
stage2.5 contract.md（权威性分级表）
        │
[2.6] scripts/match_faults.py ──► .state/fault_matches.json        （四类匹配 + 契约优先封顶 + 配额）
        │（可选）
[2.6b] templates/stage2_6_fault_match.md ──► 回写 fault_matches.json （LLM 增强：绑定/占位/contract_conflict）
        │
stage3b（条件消费）──► test_design.json（携带 fault_ref + 调和后的 oracle_refs）
        │
stage4a/4b 执行（5 分类，几乎不变）──► .state/results/*.json（sdk_defect）
        │
stage5 报告（第8部分 故障覆盖）  +  scripts/record_faults.py（仅 sdk_defect → 去重 → overlay）
        │
history_faults（闭环自积累；overlay → 人工评审 → PR 晋升全局）
```

四条贯穿原则：**文件即协议**（新产物均为文件，编排器零知识只查存在性）、**契约优先封顶**（断言级别不得超过 `contract.md` 权威性）、**优雅降级**（缺库即旧行为）、**安全收敛**（仅 `sdk_defect` 入闭环、不洗绿）。

## 三、分阶段改动详述

### Phase 1 —— 可插拔接入（matcher + stage3b 钩子）　[commit `ab07394`]

- **新增 `AutoTestFlow/scripts/match_faults.py`（阶段 2.6，纯确定性脚本）**：在 `contract.md` 就位后运行。
  - **四类触发匹配**：① 端点/方法（`applicable_scenarios` ∩ 由 contract+entry_catalog 派生的清单，RPC SUT 取宽松）；② 标签（内置 `类目/tag → specId` 映射表）；③ 历史（`history_faults.test_case_id` → 置 **P0**）；④ 关联字段（扫 `entry_catalog.params` / 场景 `data_scope` 命中 `agentId/userId/...`）。
  - **契约优先调和（封顶）**：每条故障的目标 specId 按权威性分级表给 `assert_level` 封顶——`spec-required`→至多 L2、`config-dependent`/契约静默/未知→降 L0/L1（`reconciliation=downgraded`）；保证每条用例 ≥1 条 spec-required L2 锚点。
  - **按类配额**：每场景 exception/boundary ≤3、quality ≤2；关联字段与流式 SSE 匹配**优先保留**；历史缺陷豁免配额（恒 P0）。
  - **发现约定 + 优雅降级**：`--fault-lib` 默认探测 `Specification_Repository/`；`--faults=off`/无库/无 `contract.md` → 跳过、不产出文件、exit 0。
  - 可选副产物 `.state/fault_contract_alignment.md`（替代 README 设想的 `validate_fault_contract.py`）。
- **新产物 `.state/fault_matches.json`**（"故障注入计划"）：`meta{fault_lib_version, overlay_version, stats{...}}` + `fault_matches[]{fault_id, severity, match_reason, target_endpoints, target_scenes, target_fps, branch_class, priority, trigger, expected_behavior_raw, oracle_refs[{spec_id,field,assert_level,authority,validation_point}], reconciliation, source}`。
- **`templates/stage3b_batch_design.md`**：新增"第 3.5 步（条件激活）"——若 `fault_matches.json` 存在，按 `branch_class` 映射复用既有 `case_kind`（异常/边界/质量 E2E）产出故障导向用例，**新增字段 `fault_ref`**，`oracle_refs` 直采调和结果（断言源仍是 contract.md，不上调）。
- **`AutoTestFlow/SKILL.md`**：参数表加 `--fault-lib/--fault-overlay/--faults`；阶段映射表插 `2.6 故障匹配` 行；执行流程加 stage2.5 后条件调用。

### Phase 2 —— 闭环 + 报告 + 字段修复　[commit `ab07394`，与 Phase 1 同提交]

- **新增 `AutoTestFlow/scripts/record_faults.py`（stage5 子步，纯脚本）**：**仅** `class==sdk_defect`（契约背书的真实违例）蒸馏为 `history_faults`；按 `hash(spec_id, 归一 field)` + 已知 `fault_ref` 去重；**默认 dry-run**（写 `.state/new_faults_detected.json`），`--write` 才落库；**默认写项目 overlay** 不污染全局。安全收敛：绝不收 `sut_unsatisfied/harness_defect/env_issue`。
- **修复 `AutoTestFlow/scripts/aggregate_results.py`**：原读 `sdk_bug`（与 stage4 实际写的 `sdk_defect` 不符）且计数逻辑为旧状态模型——改为读 `sdk_defect`（兼容旧 `sdk_bug`）、按 **5 分类**（passed/harness_defect/sut_unsatisfied/sdk_defect/env_issue）正确计数、透传 `fault_ref`。
- **`templates/stage5_report.md`**：新增"第 8 部分 故障库覆盖"（匹配 × 用例 × 结果，按 `fault_ref` 分组）+ `record_faults.py` 子步注记 + 输入清单/摘要表更新。
- **新增 overlay 模板 `Specification_Repository/project_faults.example.json`**：`extends`/`disabled_faults`/`fault_overrides`/`project_specific_faults`/`parameter_config`（含 `{占位符}`），落实 per-SUT 专化。

### Phase 3 —— 增强 + 保守人工门降级 + 治理（不含 MCP）　[commit `5038e56`]

- **`match_faults.py` 增强提示**：每条匹配按需打 `enrich{needs_bind, unbound_points, hints}`（validation_point 未绑定 / `downgraded` / 残留占位时置位），`meta.stats.enrichment_needed` 计数。**不改既有匹配/调和逻辑**。
- **新增 `templates/stage2_6_fault_match.md`（可选 LLM 增强子 Agent，阶段 2.6b）**：仅处理 `enrich.needs_bind` 条目——① 把自由文本 `validation_point` 绑定到最贴近 specId；② 替换 overlay 残留 `{占位符}`；③ 语义比对 `expected_behavior` vs 契约 → 置 `reconciliation:"contract_conflict"` 并降观察。**硬约束**：断言级别不得越权封顶、找不到 specId 保留观察、冲突契约优先、不洗绿。默认 `--fault-enrich off`（opt-in）。
- **保守人工门降级（`SKILL.md`）**：`fault_matches.json` 存在时，**仅 stage3b 门的"异常/边界/质量覆盖完备性"维度自动通过 + 展示摘要**；**stage1 门**与 **stage3b 的"FP 拆分映射 / 断言合理性"维度仍强制人工**（故障库只覆盖"故障判据"）。
- **治理（`Specification_Repository/README.md`）**：新增"overlay → global 晋升流程"（沉淀 overlay → 人工评审通用性/契约口径 → PR 并入全局 + bump `meta.version`）。
- **`templates/stage1_req_analyze.md`**：可选 best-effort 历史 P0 钩子（传入 `--fault-lib` 时按关键字启发式将历史缺陷关联 FP 升 P0；**权威注入仍由 2.6**）。

## 五、新增 / 变更的数据契约（schema）

| 契约/字段 | 位置 | 说明 |
|---|---|---|
| `.state/fault_matches.json` | 新产物 | 故障注入计划（见 §三 Phase 1） |

## 六、行为与兼容性

- **优雅降级（硬保证）**：所有钩子以"`.state/fault_matches.json` 存在"为条件；无库 / `--faults=off` / 纯需求模式（无 contract.md）→ 不产出、不触发，产物与接入前**字节级一致**。
- **契约安全**：① matcher 暴露前按权威性封顶 `assert_level`；② 2.6b 增强不得上调、找不到 specId 保留观察；③ 冲突→契约优先降观察；④ stage4 5 分类终极兜底——故障断言写宽只会落 `sut_unsatisfied`(skip)，绝不"洗绿"或误判缺陷。
- **人工在环**：保守降级仅自动化"故障覆盖完备性"维度；FP 拆分与断言合理性仍人工。

## 八、已知限制 / 延后项

- **MCP 作源**：`05` §b 推荐的"file 核心 + MCP 作源（起点物化版本化快照）"混合方案**按决定本轮不实现**（环境无 MCP 服务器，客户端不可测）。

- **stage1 历史 P0 钩子**：best-effort 启发式（`test_case_id` 与当前 FP 无硬映射）；权威注入在 2.6。
- **真实 SUT 端到端**：stage2.6→3b→4→5 在真实 spring-ai-ascend 上的运行**未执行**（按用户要求跳过）。
- **提交 Verified 状态**：当前环境缺私有签名密钥（仅 `.pub`）且签名代理只读，提交在 GitHub 显示 **Unverified**；不影响内容正确性。

## 十、使用指引（启用 / 复现）

```bash
# 启用（标准模式自动探测 Specification_Repository/）
... AutoTestFlow ... --fault-lib Specification_Repository/rest_api_common_faults.json [--faults auto] [--fault-overlay <proj>/fault_library/project_faults.json] [--fault-enrich on]

# 复现自测（无需 SUT）
python AutoTestFlow/scripts/match_faults.py   --output-dir AutoTestFlow/examples/a2a/fault_demo --fault-lib Specification_Repository/rest_api_common_faults.json
python AutoTestFlow/scripts/aggregate_results.py --output-dir AutoTestFlow/examples/a2a/fault_demo
python AutoTestFlow/scripts/record_faults.py  --output-dir AutoTestFlow/examples/a2a/fault_demo --overlay-path AutoTestFlow/examples/a2a/fault_demo/project_faults.json --write
```

- 关闭：`--faults off`（与未接入一致）。
- per-SUT：复制 `Specification_Repository/project_faults.example.json` 为各项目 `fault_library/project_faults.json`，`--fault-overlay` 指定。
- 详见 `AutoTestFlow/examples/a2a/fault_demo/README.md` 与 `FaultsAnalysis/04` 验证计划。
