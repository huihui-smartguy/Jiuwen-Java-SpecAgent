# v6.0 Stage2：通用代码扫描 Adapter Layer

## 背景

原 Stage2 prompt 实际是 Java/Spring 专用静态扫描流程，强依赖 Java 文件、Spring 注解、ControllerAdvice 和 Java 序列化模式。被测对象如果是 Python、C++ 或其他技术栈，Stage2 难以稳定产出供 Stage2.5 与 Stage3a 使用的代码事实。

## 三轮深入分析

1. 问题不只在 `templates/stage2_code_scan.md`：`SKILL.md`、`README.md`、`DESIGN.md`、`scenario_schema.md`、`code_analysis_template.md` 也把 Stage2 描述成 Java/Spring-only。
2. 下游事实模型应保持稳定：继续保留 `entry_catalog`、`exception_catalog`、`constraint_catalog`、`serialization_facts`、`code_defects`、`code_only_capabilities` 与 `framework_scenes`。
3. 语言差异应下沉到 adapter layer：先做确定性 profile 预扫描，再由 Stage2 Agent 按 profile 执行 P0/P1/P2 探测。

## 主要变更

- 新增 `scripts/prepare_code_scan.py`，检查 `code_path` 并写入 `.state/code_scan_plan.json`。
- 新增 `shared/code_scan_profiles.json`，首批支持 `java.spring`、`python.web_api`、`cpp.service_rpc`、`generic.source_tree`。
- 新增 `shared/code_scan_guide.md` 作为 Stage2 通用指南；`java_scan_guide.md` 调整为 Java/Spring profile 附录。
- 重写 `templates/stage2_code_scan.md`，改为读取 `code_scan_plan.json` 并按 profile 分派扫描策略。
- 泛化 README、DESIGN、SKILL、schema 与代码分析模板中的 Stage2 表述。

## 兼容策略

- 保留现有 `s2_code_facts.json` 必填字段，不删除旧字段。
- 新增 profile 元数据均为可选：`meta.primary_profile`、`meta.language`、`meta.frameworks`、`meta.profile_confidence`、`meta.scan_plan`。
- `entry_catalog[]` 可选新增 `transport`、`framework`、`evidence`。
- `framework_scenes.json.meta` 可选新增 `source_profile`。
- `contract.md` 仍是唯一强断言 oracle；源码事实只作为静态线索。

## 验证

- 新增 `tests/test_stage2_code_scan_adapters.py`，覆盖 Java/Spring、Python Web/API、C++ service/RPC 与 generic fallback。
- 校验每个 profile 都具备必需 probe category。
- 增加文档守卫，防止 Stage2 再退回“全局 Java/Spring 假设”。
