# v2.4 AutoTestFlow Fault Oracle Trace Gate / 故障 Oracle 轨迹门禁

## 中文

### 背景

本次优化修复一个关键缺陷：故障库驱动的异常用例如果最终 E2E 执行成功，过去可能被直接判为通过，导致中间交互过程中的 5xx、错误帧、重复终态事件、id 未回带、资源/状态副作用等缺陷被隐藏。

### 变更

- 将故障库用例明确为可执行缺陷探针，而不是带异常输入的普通 E2E 用例。
- `match_faults.py` 为每条命中补齐 `fault_oracles`，覆盖结果、过程和否定期望，并保证至少有一条 required 的过程/否定 oracle。
- Stage3b 要求每个 `fault_ref` 用例原样保留 `fault_oracles`；缺失 required 过程/否定 oracle 视为设计错误。
- 新增 `scripts/evaluate_fault_oracles.py`，在 Stage4 pytest 通过后读取 `.state/trace/*.jsonl`，确定性检查 required fault oracle。
- Stage4 模板更新分类顺序：就绪门和 harness 错误优先，其次 pytest 断言失败，再执行 fault oracle 门禁，最后才允许 `passed`。
- `aggregate_results.py` 汇总 `fault_oracle_summary` / `fault_oracle_results`，并统计 “pytest 已通过但 fault oracle 阻断” 的用例。
- Stage5 报告新增 Fault Oracle Coverage，展示 outcome/process/negative oracle 的 pass/fail/unobservable 统计。

### 安全规则

- `fault_ref` 用例不能仅凭最终响应成功写 `status=passed`。
- required fault oracle 失败时，依据 spec-required/fault-required 证据归入 `sdk_defect`；否则归入 `sut_unsatisfied`。
- required fault oracle 不可观察或 unsupported 时归入 `requires_human_review`，不得静默通过。
- `record_faults.py` 仍只沉淀确认的 `sdk_defect`，包括由 fault oracle trace 证据确认的中间交互缺陷。
- remediation / Stage6 保持门控：只有确认缺陷和足够证据才进入可修复子集，外发 issue 仍需人工确认。

## English

### Background

This update fixes a critical blind spot: fault-library cases could previously be marked as passed when the final E2E execution succeeded, even if intermediate interactions contained defects such as 5xx responses, error frames, duplicate terminal events, broken id correlation, or unintended state/resource mutation.

### Changes

- Treat fault-library cases as executable defect probes rather than normal E2E cases with abnormal inputs.
- `match_faults.py` now normalizes `fault_oracles` for every match, covering outcome, process, and negative expectations, with at least one required process/negative oracle.
- Stage3b requires every `fault_ref` case to preserve `fault_oracles`; missing required process/negative oracles are design errors.
- Added `scripts/evaluate_fault_oracles.py` to deterministically evaluate required fault oracles from `.state/trace/*.jsonl` after pytest succeeds.
- Stage4 templates now classify in order: readiness/harness issues, pytest assertion failures, fault-oracle gate, and only then `passed`.
- `aggregate_results.py` carries `fault_oracle_summary` / `fault_oracle_results` and counts cases where pytest passed but fault oracles blocked the pass.
- Stage5 reporting adds Fault Oracle Coverage for outcome/process/negative oracle pass/fail/unobservable counts.

### Safety Rules

- A `fault_ref` case cannot write `status=passed` on final response success alone.
- Required fault-oracle failures become `sdk_defect` when backed by spec-required/fault-required evidence; otherwise they become `sut_unsatisfied`.
- Required unsupported or unobservable fault oracles become `requires_human_review`, never silent pass.
- `record_faults.py` still records only confirmed `sdk_defect`, now including intermediate trace-backed defects.
- remediation / Stage6 remains gated: only confirmed, evidence-backed defects enter the patchable subset, and external issue submission still requires human confirmation.
