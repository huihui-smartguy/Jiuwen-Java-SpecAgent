# v1.2 - TestKnowledgeBase Runtime Registry

Date: 2026-06-29

## Background

`Specification_Repository` is a legacy Java/Spring-oriented fault repository and should not be a practical dependency for future AutoTestFlow iterations. TestKnowledgeBase is the forward source of truth, but v1.1 only exposed individual JSON files and did not provide a machine-readable runtime contract for discovery or matching.

## Changes

- Added `TestKnowledgeBase/registry.json` as the runtime package registry.
- Registered REST/API, Web, Agent, DFX, and professional acceptance packages.
- Moved package-level matching hints, branch classification, contract hint kinds, and recording policy into TestKnowledgeBase metadata.
- Updated TestKnowledgeBase version to `1.2.0`.
- Updated TestKnowledgeBase manifest and transformation scope to make registry discovery the normal runtime path.

## Runtime Policy

- AutoTestFlow should read `TestKnowledgeBase/registry.json` by default.
- New fault packages should be added to the registry instead of hard-coded in AutoTestFlow.
- `Specification_Repository` may be used only by explicit legacy invocations.
- Strong executable assertions still require `contract.md`; TestKnowledgeBase can prioritize and suggest coverage but must not become an oracle by itself.

## Validation Focus

- JSON parseability.
- Unique fault IDs across all TestKnowledgeBase fault packages.
- Registered package paths exist.
- Registered fault packages expose `domain`, matching rules, and contract hint policy.
