# v6.0 Stage 2: Generalized Code Scan Adapter Layer

## Background

The previous Stage 2 prompt was effectively a Java/Spring static-analysis workflow. It required Java files, Spring annotations, controller advice, and Java serialization patterns, so Python, C++, and other SUTs could not produce reliable code facts for Stage 2.5 and Stage 3a.

## Three-Round Analysis

1. The issue was broader than `templates/stage2_code_scan.md`: `SKILL.md`, `README.md`, `DESIGN.md`, `scenario_schema.md`, and `code_analysis_template.md` also described Stage 2 as Java/Spring-only.
2. The downstream fact model should remain stable: `entry_catalog`, `exception_catalog`, `constraint_catalog`, `serialization_facts`, `code_defects`, `code_only_capabilities`, and `framework_scenes`.
3. Language-specific discovery should move into an adapter layer with deterministic preflight detection and profile-specific scan hints.

## Main Changes

- Added `scripts/prepare_code_scan.py`, which inspects `code_path` and writes `.state/code_scan_plan.json`.
- Added `shared/code_scan_profiles.json` with v1 profiles: `java.spring`, `python.web_api`, `cpp.service_rpc`, and `generic.source_tree`.
- Added `shared/code_scan_guide.md` as the language-neutral Stage 2 guide; `java_scan_guide.md` is now the Java/Spring profile appendix.
- Rewrote `templates/stage2_code_scan.md` to consume `code_scan_plan.json` and dispatch profile-specific P0/P1/P2 probes.
- Generalized Stage 2 wording in README, DESIGN, SKILL, schema, and code-analysis template.

## Compatibility

- Existing `s2_code_facts.json` fields are preserved.
- New profile metadata is optional: `meta.primary_profile`, `meta.language`, `meta.frameworks`, `meta.profile_confidence`, and `meta.scan_plan`.
- `entry_catalog[]` may include optional `transport`, `framework`, and `evidence`.
- `framework_scenes.json.meta` may include optional `source_profile`.
- `contract.md` remains the only strong oracle for executable assertions.

## Verification

- Added `tests/test_stage2_code_scan_adapters.py` for Java/Spring, Python Web/API, C++ service/RPC, and generic fallback detection.
- Added profile schema coverage checks for all required probe categories.
- Added documentation guard checks to prevent Stage 2 from regressing to a universal Java/Spring assumption.
