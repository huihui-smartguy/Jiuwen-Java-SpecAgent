# v2.3 AutoTestFlow 默认知识库增强与门控修复命令 / Default Knowledge-Enriched Remediation Command

## 中文

### 背景

本次调整把 AutoTestFlow 的推荐入口升级为完整闭环命令：默认接入 `TestKnowledgeBase`、默认启用故障库 LLM 增强，并默认进入门控式 fault analysis / remediation。用户运行前只需按项目实际情况修改需求文档、SUT manifest 和 `remediation.config.json`。

### 主要变更

- 在 `AutoTestFlow/README.md` 最显著位置新增默认全链路命令：
  `/auto-test-flow requirements.md --sut-manifest autotestflow.suts.md --remediation-config remediation.config.json --faults on --fault-enrich on --remediate on`
- 更新 `AutoTestFlow/SKILL.md` 的编排默认值：`--faults=on`、`--fault-enrich=on`、`--remediate=on`。
- `--faults=on` 默认读取 `TestKnowledgeBase/registry.json`，并继续支持通过 `--faults off` 显式关闭。
- `--fault-enrich=on` 默认在存在待绑定故障项时运行 stage2.6b，执行 specId 模糊绑定、占位符替换和 `contract_conflict` 检出。
- `--remediate=on` 默认在 stage5 后尝试进入 fault analysis / remediation，但必须满足 remediation 配置就位、存在可分析目标、人工确认门通过等条件。

### 安全纪律

- 缺少或非法 `remediation.config.json` 时，stage6/7 明确跳过并记录原因，不强行修复。
- evidence issue 外发仍必须同时满足：`--remediate=on`、stage6 人工确认门通过、`switches.allow_open_issue=true`、`require_evidence_before_issue=true`。
- 继续保持 issue-only：不推送 fork、不创建 PR、不读取或输出 token。
- `contract.md` 仍是唯一强 Oracle；故障库与 wiki 只作为 advisory 知识源。

## English

### Background

This update promotes AutoTestFlow's recommended entrypoint to a full closed-loop command. It enables `TestKnowledgeBase`, LLM-assisted fault enrichment, and gated fault analysis / remediation by default. Before running it, users only need to adjust the requirements document, SUT manifest, and `remediation.config.json` for their project.

### Main Changes

- Added the default full-pipeline command to the most prominent area of `AutoTestFlow/README.md`:
  `/auto-test-flow requirements.md --sut-manifest autotestflow.suts.md --remediation-config remediation.config.json --faults on --fault-enrich on --remediate on`
- Updated `AutoTestFlow/SKILL.md` orchestration defaults to `--faults=on`, `--fault-enrich=on`, and `--remediate=on`.
- `--faults=on` reads `TestKnowledgeBase/registry.json` by default, while `--faults off` remains the explicit disable path.
- `--fault-enrich=on` runs stage2.6b when fault matches need binding, covering specId binding, placeholder substitution, and `contract_conflict` detection.
- `--remediate=on` attempts gated fault analysis / remediation after stage5, but only when remediation config exists, actionable targets are present, and required gates pass.

### Safety Rules

- Missing or invalid `remediation.config.json` causes stage6/7 to skip with a clear reason; AutoTestFlow must not force a repair.
- Evidence issue submission still requires all gates: `--remediate=on`, the stage6 human confirmation gate, `switches.allow_open_issue=true`, and `require_evidence_before_issue=true`.
- The flow remains issue-only: no fork push, no PR creation, and no token exposure.
- `contract.md` remains the only strong oracle; fault libraries and wiki material are advisory knowledge sources.
