| 项 | 值 |
|---|---|
| 版本 | **v2.0** |
| 日期 | 2026-06-22 |
| 分支 | `claude/charming-edison-ubgj86` |

> 本记录总结 v2.0「自动修复」特性。设计与红线见 `AutoTestFlow/DESIGN.md` §7 与 `AutoTestFlow/shared/remediation_rules.md`；

---

## 一、背景与动机

- v1.x 流水线到 **stage5 报告**即止：`sdk_defect`（`contract.md` 背书的真实违例）只是被"记录"，仍需人工读报告、定位代码、修改、提 PR、开 issue。
- 需求：把这一段**闭环自动化**——对"测试不通过"中的 `sdk_defect`，深度分析并修复开发仓**业务代码** + 开发仓**自测代码（回归）**，自动提交 PR；同时同步提交 bug **issue**，讲清"根据规格库哪些知识 + 哪些实测结果，推断该 bug 成立"。
- 该能力外发副作用强（改外部仓、开 PR/issue），且放大了 `DESIGN.md` §2「生成器不得自我认证」的风险，故核心是**门控 + 不自我认证 + 仅 sdk_defect**。
- **用户决策（锁定）**：① 仅 `sdk_defect` 触发；② 修复后**本地重建 + 重跑失败用例转绿**再开 PR；③ **fork→upstream PR** + upstream issue；④ **配置文件**声明 SUT 与代码仓；⑤ 开关 + 强制人工确认。

---

## 二、总体架构（数据流）

```
stage5 (report.md)
  │  [进入条件: --remediate≠off ∧ remediation.config 就位 ∧ case_results.summary.sdk_defects>0]
  ├─ stage6 缺陷深析（只读，每个 sdk_defect 一个子Agent）
  │     remediation_plan.py(prep) → plan.json
  │     stage6_defect_analyze.md  → defects/<case>/{root_cause.md,patch.diff,regression_test.diff,issue.md,pr.md,confidence.json}
  │     remediation_plan.py(--finalize) → manifest.json
  │  ── 【强制人工确认门 AskUserQuestion，仅 --remediate=on】读 manifest 展示摘要 ──
  └─ stage7 应用+构建+复验+提交（确定性脚本，副作用）
        apply_and_reverify.py: clone@ref→分支→git apply(路径白名单)→build→重启SUT→重跑失败用例(须转绿)+回归自测 → reverify.json
        submit_remediation.py: push fork → gh PR(fork→upstream) → gh issue(upstream) → submitted.json
```

四条贯穿原则（承接 v1.x 内核）：**契约唯一权威**（修 SUT 符合 `contract.md`）、**不自我认证**（绿由脚本重跑判定）、**文件即协议 / 零知识门控**（门只读 `manifest.json` 小文件）、**优雅降级**（关闭即旧行为）。

## 三、分阶段改动

### stage6 —— 缺陷深度分析（只读，产工件）

- 前处理 `remediation_plan.py`：读 `case_results.json` + `.state/results/*.json`，**仅过滤 `class==sdk_defect`**（承接 `record_faults.py` 的收敛口径），按 `--remediation-max-defects` 产 `plan.json` worklist。
- 子 Agent `stage6_defect_analyze.md`（每缺陷一个）：自读 `contract.md`（§7 权威性）+ `results/<case>.json` + `trace/<case>.jsonl` + `fault_matches.json` + `s2_code_facts.json` + 克隆源码，产出 `root_cause.md` / `patch.diff`(仅 `business_code_roots`) / `regression_test.diff`(仅 `selftest_roots`，新增) / `issue.md`(**两段证据**) / `pr.md` / `confidence.json`。
- 后处理 `remediation_plan.py --finalize`：汇 `confidence.json` → `manifest.json`（门面小文件）。

### 强制人工确认门（仅 `--remediate=on`）

- 提交前必经 `AskUserQuestion`，读 `manifest.json` 展示每缺陷 `spec_id / files_touched / ±行 / confidence / issue_title`；选项：确认提交 / 仅 dry-run / 仅记 issue 不提 PR / 取消。**任何 clone-push/PR/issue 在门之后**。

### stage7 —— 应用 + 构建 + 复验 + 提交（确定性脚本，无 LLM）

- `apply_and_reverify.py`（**绿门**，dry-run 与 on 均执行）：幂等 clone/物化 @ref → `checkout -B 分支` → `git apply`（路径白名单）→ `build_cmd` → stop/restart SUT + 就绪轮询 → 重跑失败的 `test_<case>.py`（指向重建后的 SUT，须转绿）+ `selftest_cmd` → `reverify.json{remediated[],still_red[],before/after}`。失败全落盘、剔除该缺陷、不抛栈。
- `submit_remediation.py`（仅 `on`+过门+`switches.allow_*`+green）：push fork → `gh pr create`(fork→upstream，含 before/after，默认 draft) → `gh issue create`(upstream，逐缺陷)；幂等去重 → `submitted.json`。`dry-run`/`allow_*=false` → **零 gh 调用**。

## 六、行为与兼容性

- **优雅降级（硬保证）**：`--remediate=off`（默认）/ 无配置 / 无 sdk_defect → 不进入 stage6/7、不产 `.state/remediation/`，流水线与 v1.x **字节级一致**。
- **不自我认证**：绿/红由 `apply_and_reverify.py` 重跑**未改动**用例 + **重建后** SUT 判定；LLM 不判绿。
- **双层开关**：动作执行 ⟺ `--remediate=on`（+过门）**且** `switches.allow_*=true`；`require_green_before_pr` 不可关闭。
- **不入库**：克隆出的代码仓 `**/.state/remediation/repo/`、复验轨迹 `trace/` 经 `.gitignore` 排除。

## 七、验证（离线确定性，无需真实 SUT/仓/gh）

基于 `examples/a2a/remediation_demo/`（内置 `fake_repo` + 两处 seeded bug + `reverify.simulate=true` + no-op 构建/运行）：

1. `remediation_plan.py`(prep) → `plan.json` 恰含 2 个 sdk_defect（TC_041 SPEC-RESP-WRAP / TC_042 SPEC-ID-TYPE）。
2. `--finalize` → `manifest.json`：2 defects，localizable 2，needs_human 0。
3. `apply_and_reverify.py`：`repo_method=materialized`，两条 `patch.diff` **真实 `git apply` 成功** + 回归补丁应用，`remediated=[TC_041,TC_042]`、`still_red=[]`，分支 `autotestflow/fix/TC_041-TC_042`。
4. `submit_remediation.py --remediate dry-run` → **零 gh 调用**，`submitted.json{mode:dry-run,skipped}`。
5. **第二层保险**：`--remediate on --gate-confirmed` 但 `allow_*=false` → 仍零 gh，push/pr/issue 三跳过。
6. **绿前提红线**：`require_green_before_pr=false` → 配置加载即 `ConfigError`。
7. **路径白名单**：业务补丁越出 `business_code_roots` → 拒绝。
8. **幂等**：`apply_and_reverify.py` 连跑两次 → `reverify.json` 字节一致。
9. 四个 Python 文件 `py_compile` 通过。

- **诚实说明**：真实 `clone + mvn/gradle 构建 + SUT 重启 + pytest 重跑 + gh PR/issue` 仅在用户用**真实仓 + 构建工具链 + gh 鉴权**运行 `--remediate=on/dry-run` 时走通；本环境用 `simulate=true` 离线验证编排与产物形态（与 README §9 口径一致）。

## 八、已知限制 / 延后

- 真实 SUT 端到端（克隆 spring-ai-ascend@v0.1.0 + 构建 + 重启 + gh）**未在本环境运行**（无可达 SUT / Java 工具链 / 用户 fork 鉴权）。
- stage6 的代码定位与补丁质量依赖 LLM；定位不到 → `needs_human`、仅记 issue 不改码。
- 单轮影响面受 `--remediation-max-defects`（默认 5）限制；PR 默认 draft，需人工合并。
- 提交在 GitHub 显示 **Unverified**（环境缺私有签名密钥，已知限制）。

## 九、使用指引

```bash
# 启用（标准模式 + 自动修复；经强制确认门后真实提交）
/auto-test-flow 需求.md <java仓>/<模块> --sut-base-url http://host:port \
    --remediate on --remediation-config <proj>/remediation.config.json

# 仅本地复验、不外发
... --remediate dry-run

# 复现离线 demo（无需 SUT/仓/gh）
python AutoTestFlow/scripts/remediation_plan.py   --output-dir AutoTestFlow/examples/a2a/remediation_demo
python AutoTestFlow/scripts/remediation_plan.py   --output-dir AutoTestFlow/examples/a2a/remediation_demo --finalize
python AutoTestFlow/scripts/apply_and_reverify.py --output-dir AutoTestFlow/examples/a2a/remediation_demo
python AutoTestFlow/scripts/submit_remediation.py --output-dir AutoTestFlow/examples/a2a/remediation_demo --remediate dry-run
```

- 关闭：`--remediate off`（默认，与未接入一致）。
- per-SUT：复制 `AutoTestFlow/examples/remediation.config.example.json`，按 `shared/remediation_config_schema.md` 改 `repo.fork_url` 等。
- 详见 `AutoTestFlow/examples/a2a/remediation_demo/README.md` 与 `AutoTestFlow/shared/remediation_rules.md`。
