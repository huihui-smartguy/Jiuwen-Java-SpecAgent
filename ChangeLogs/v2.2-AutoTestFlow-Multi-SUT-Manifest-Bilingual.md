# v2.2 AutoTestFlow Multi-SUT Manifest / 多 SUT Manifest 调用重构

## 中文

### 背景与动机

原有调用方式依赖单个 `--sut-base-url`，只能表达一个运行中的被测对象。实际测试中，一个需求可能涉及多个 SUT：它们可能互为依赖、互相独立，或需要由 AutoTestFlow 在测试前尝试构建和启动。单 URL 模型会把契约、就绪门、执行结果和修复配置混在同一个输出空间中，容易造成 oracle 污染。

### 主要变化

- 新增标准入口：`/auto-test-flow 需求.md --sut-manifest autotestflow.suts.md`。
- 新增 Markdown + YAML manifest schema：`autotestflow.sut-manifest.v1`。
- 每个 target 独立输出到 `targets/<target_id>/`，独立拥有 `contract.md`、`case_results.json`、`.state/results/`、`.state/trace/` 和 target 报告。
- 新增 `scripts/sut_manifest.py`：解析 front matter、校验 target id / 依赖关系 / runtime 字段、归一化 source 和 output path，并写 `.state/sut_manifest.normalized.json`。
- 新增 `scripts/sut_runtime.py`：执行 target readiness；managed runtime 的 build/start/stop 命令必须显式传入 `--allow-commands`，用于承接人工确认门。
- `probe_contract.py` 支持 `--target-id` 和 `--probe-plan`，保留现有 A2A 示例探针作为 legacy fallback。
- `reference/http_client.py` 与 `reference/conftest.py` 优先使用 `AUTOTESTFLOW_TRACE_DIR`，继续兼容 `A2A_TRACE_DIR`。

### 兼容性

- `--sut-base-url` 保留为 deprecated legacy shim；内部归一化为 `target_id=default` 的单 target manifest。
- TestKnowledgeBase registry 行为不变；知识库仍决定“测什么/如何优先级”，强断言仍只能来自 target-local `contract.md`。
- 全局 `--remediation-config` 仅用于 legacy 单 target；multi-SUT 优先使用 manifest target 的 `remediation.config_path` 或 target-local `remediation.config.json`。

### 验证

- 新增 manifest 解析测试：双 target、重复 id、缺少 runtime 字段、非法依赖、legacy `--sut-base-url` 归一化。
- 新增 probe 测试：`--target-id` 元数据、manifest probe plan dry-run、不可达服务 graceful fallback、legacy fallback。
- 新增 runtime 测试：predeployed readiness、managed command dry-run/gating、target-local env_issue 隔离。
- 扩展文档守卫：README/SKILL 不再把 `--sut-base-url` 展示为主调用方式。

## English

### Motivation

The previous invocation relied on a single `--sut-base-url`, which can only describe one running SUT. Real-world test plans often involve multiple systems: related services, independent services, or services that AutoTestFlow may need to build and start before testing. A single-URL model risks mixing contracts, readiness state, execution results, and remediation config into one output space.

### Key Changes

- Added the standard invocation: `/auto-test-flow requirements.md --sut-manifest autotestflow.suts.md`.
- Added a Markdown + YAML manifest schema: `autotestflow.sut-manifest.v1`.
- Each target writes to `targets/<target_id>/` and owns its own `contract.md`, `case_results.json`, `.state/results/`, `.state/trace/`, and target report.
- Added `scripts/sut_manifest.py` to parse front matter, validate target ids / dependencies / runtime fields, normalize source and output paths, and write `.state/sut_manifest.normalized.json`.
- Added `scripts/sut_runtime.py` for target readiness and managed runtime build/start/stop command gating. Commands require explicit `--allow-commands`, matching the human-confirmation gate.
- Updated `probe_contract.py` with `--target-id` and `--probe-plan`, while preserving the existing A2A example probes as the legacy fallback.
- Updated `reference/http_client.py` and `reference/conftest.py` to prefer `AUTOTESTFLOW_TRACE_DIR` while continuing to honor `A2A_TRACE_DIR`.

### Compatibility

- `--sut-base-url` remains as a deprecated legacy shim and normalizes to a single target with `target_id=default`.
- TestKnowledgeBase registry behavior is unchanged. The knowledge base decides what to test and how to prioritize; strong assertions still come only from the target-local `contract.md`.
- Global `--remediation-config` remains valid for legacy single-target runs only. Multi-SUT runs prefer manifest target `remediation.config_path` or target-local `remediation.config.json`.

### Verification

- Added manifest parsing tests for a valid two-target manifest, duplicate ids, missing runtime fields, invalid dependencies, and legacy `--sut-base-url` conversion.
- Added probe tests for target metadata, manifest probe-plan dry-run, unreachable fallback, and legacy fallback.
- Added runtime tests for predeployed readiness, managed command dry-run/gating, and target-local env_issue isolation.
- Extended documentation guards so README/SKILL no longer present `--sut-base-url` as the primary invocation.
