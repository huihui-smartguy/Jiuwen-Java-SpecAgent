## 中文

### AutoTestFlow v2.0.0

AutoTestFlow v2.0.0 将 v1.0 的单 SUT、REST/API 倾向流程升级为通用测试智能体管线。新的默认入口围绕 `autotestflow.suts.md`、`remediation.config.json` 与 `TestKnowledgeBase/registry.json` 组织完整闭环：需求分析、SUT 建模、代码扫描、知识匹配、用例设计、执行判定、报告、故障分析和门控修复。

主要创新：

- Multi-SUT manifest 成为标准入口，每个 target 独立管理契约、执行结果、trace、报告和修复配置，避免跨系统 oracle 污染。
- `autotestflow.suts.md` 支持自然语言 SUT 描述，并保留 Markdown+YAML manifest 兼容；解析产物记录置信度、复核原因和敏感信息脱敏。
- Stage 2 从 Java/Spring 专用扫描升级为 profile adapter：支持 `java.spring`、`python.web_api`、`cpp.service_rpc` 与通用源码树，并保持下游事实模型稳定。
- 默认接入 TestKnowledgeBase registry，支持 REST/API、Web、Agent、DFX 等多域故障知识；知识库决定测试优先级，强断言仍只来自 `contract.md`。
- 故障库用例升级为可执行缺陷探针，新增 fault oracle trace gate，防止最终响应成功掩盖中间 5xx、错误帧、重复终态、id 未回带或副作用缺陷。
- Stage 1 测试建议成为一等可追踪对象，用例必须引用建议或显式标记不适用，提升需求建议到测试覆盖的可证明性。
- 输出归档迁移到 `RunMetadata/`、`FeatureAnalysis/`、`Contract/`、`KnowledgeBase/`、`QualityGates/`、`TestCases/`、`TestRun/`、`Reports/`、`FaultAnalysis/`、`Remediation/` 等规范目录，同时保留旧归档只读 fallback。
- 引入 Professional Acceptance Phase A-D，为测试计划、用例设计、发布门禁、残余风险、AI/Agent readiness 和报告质量提供 advisory gate。
- Stage 6/7 从自动 PR 修复收敛为安全的 fault analysis / issue-only 闭环：只有 spec-required 且可安全修复的缺陷进入 patch/reverify 子集，外发 evidence issue 必须通过人工确认和配置门。
- 新增 Markdown 伴生产物与 Quickstart starter，让人工复核、LLM 阅读和新用户入门更直接。

### TestKnowledgeBase v2.0.0

TestKnowledgeBase v2.0.0 正式成为后续迭代的前向知识源，替代 `Specification_Repository` 的隐式运行时地位。它不仅保存故障模式，也为 AutoTestFlow 提供可发现、可校验、可扩展的运行时知识合约。

主要创新：

- 新增并主干化 `TestKnowledgeBase/registry.json`，作为 AutoTestFlow 默认运行时发现入口。
- 将 REST/API、Web Frontend、Agent Scene、DFX Reliability 和 Professional Acceptance 注册为可独立演进的知识包。
- 将 domain、branch classification、contract hint kinds、scenario signals、recording policy 等匹配元数据放入知识库，减少 AutoTestFlow 内部硬编码。
- 明确知识库契约边界：知识可建议覆盖、优先级和报告内容，但强可执行 oracle 必须引用 `contract.md`。
- `Professional_experience` 新增 runtime 字段和 schema，支持专业验收 Phase A-D，同时保持 advisory-only。
- 新增候选知识沉淀与 promotion 策略，AutoTestFlow 只自动记录确认的 `sdk_defect` 项目候选，进入正式知识包前仍需评审。
- 知识库校验覆盖 JSON 可解析性、fault id 唯一性、registry path 存在性、runtime 字段和专业验收声明。

### 兼容性与安全

- Legacy `--sut-base-url`、旧 YAML manifest、旧输出归档读取路径和 `.state/fault_matches.json` 兼容输出继续保留。
- `Specification_Repository` 仅保留为显式 legacy 输入，不再作为默认 fallback。
- 缺少或非法 remediation 配置时跳过 Stage 6/7，不强行修复。
- 默认 Quickstart 的 `allow_open_issue=false`，不会自动外发 issue。
- 全流程不推送 fork、不创建 PR、不读取或输出 token。

## English

### AutoTestFlow v2.0.0

AutoTestFlow v2.0.0 upgrades the v1.0 single-SUT, REST/API-oriented flow into a generalized test-intelligence pipeline. The default entrypoint now centers on `autotestflow.suts.md`, `remediation.config.json`, and `TestKnowledgeBase/registry.json` to orchestrate requirements analysis, SUT modeling, code scanning, knowledge matching, case design, execution verdicts, reporting, fault analysis, and gated remediation.

Key innovations:

- Multi-SUT manifests are now the standard invocation model. Each target owns its contract, execution results, traces, reports, and remediation config, preventing cross-system oracle contamination.
- `autotestflow.suts.md` supports natural-language SUT descriptions while preserving Markdown+YAML manifest compatibility; parse artifacts record confidence, review reasons, and secret redaction.
- Stage 2 moved from a Java/Spring-specific scan to a profile adapter layer for `java.spring`, `python.web_api`, `cpp.service_rpc`, and generic source trees while preserving the downstream fact model.
- TestKnowledgeBase registry is used by default across REST/API, Web, Agent, and DFX domains. Knowledge guides coverage and priority, while `contract.md` remains the only source of strong executable assertions.
- Fault-library cases are now executable defect probes with a fault-oracle trace gate, preventing final response success from hiding intermediate 5xx responses, error frames, duplicate terminal events, broken id correlation, or side effects.
- Stage 1 test suggestions are first-class traceable objects; cases must reference each suggestion or explicitly mark it not applicable.
- Output archives now use canonical directories such as `RunMetadata/`, `FeatureAnalysis/`, `Contract/`, `KnowledgeBase/`, `QualityGates/`, `TestCases/`, `TestRun/`, `Reports/`, `FaultAnalysis/`, and `Remediation/`, with read-only fallback for legacy archives.
- Professional Acceptance Phase A-D adds advisory gates for planning, case design, release readiness, residual risk, AI/Agent readiness, and report quality.
- Stage 6/7 is now a safer fault-analysis / issue-only loop: only spec-required and safely patchable defects enter patch/reverify, and evidence issue submission requires both human confirmation and config gates.
- Markdown companion artifacts and Quickstart starters make human review, LLM review, and first-run onboarding easier.

### TestKnowledgeBase v2.0.0

TestKnowledgeBase v2.0.0 is the forward knowledge source for future iterations, replacing the implicit runtime role of `Specification_Repository`. It now provides not only fault patterns, but also a discoverable, validated, extensible runtime knowledge contract for AutoTestFlow.

Key innovations:

- Added and promoted `TestKnowledgeBase/registry.json` as AutoTestFlow's default runtime discovery entrypoint.
- Registered REST/API, Web Frontend, Agent Scene, DFX Reliability, and Professional Acceptance as independently evolvable knowledge packages.
- Moved domain, branch classification, contract hint kinds, scenario signals, and recording policy metadata into the knowledge base, reducing AutoTestFlow hard-coding.
- Clarified the contract boundary: knowledge can suggest coverage, priority, and report content, but strong executable oracles must cite `contract.md`.
- Added Professional Experience runtime fields and schema for Professional Acceptance Phase A-D while keeping the layer advisory-only.
- Added candidate knowledge recording and promotion policy. AutoTestFlow records only confirmed `sdk_defect` project candidates automatically; promotion into official packages remains review-gated.
- Knowledge validation covers JSON parseability, unique fault ids, registry paths, runtime fields, and professional acceptance declarations.

### Compatibility And Safety

- Legacy `--sut-base-url`, existing YAML manifests, legacy archive read paths, and `.state/fault_matches.json` compatibility output remain supported.
- `Specification_Repository` remains available only for explicit legacy inputs and is no longer a default fallback.
- Missing or invalid remediation config skips Stage 6/7 instead of forcing repair.
- The default Quickstart keeps `allow_open_issue=false`, so it will not publish issues automatically.
- The flow does not push forks, create PRs, read tokens, or print tokens.
