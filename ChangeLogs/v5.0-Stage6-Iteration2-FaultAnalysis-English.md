# v5.0 Stage 6 Iteration 2: Fault Analysis and Evidence Issues

## Background

The previous Stage 6 flow only generated remediation artifacts for `sdk_defect` cases. That was too narrow for the Web, REST API, Agent, DFX, and future failure-mode libraries under `TestKnowledgeBase/Fault`. This iteration upgrades Stage 6 into an extensible `fault_analysis` layer while preserving the safe patch/reverify subset.

## Main Changes

- Added `.state/fault_analysis/analysis_plan.json` and `.state/fault_analysis/manifest.json` for patchable, issue-only, and diagnostic analysis targets.
- Added `shared/fault_analysis_profiles.json` so Web / REST / Agent / DFX / Security / Observability are data-driven profiles, with a default profile for future failure modes.
- Kept `.state/remediation/plan.json` as the Stage 7 compatibility subset for spec-required, safely patchable contract defects only.
- Added `templates/stage6_fault_analyze.md` for non-patchable and future-domain targets. It outputs `evidence.json`, `root_cause.md`, `fix_solution.md`, `issue.md`, and `confidence.json`.
- Updated `templates/stage6_defect_analyze.md` and the beta template so patchable targets also produce evidence / fix-solution artifacts and no longer reference PRs.
- Removed automated PR submission from Stage 7. The flow no longer pushes fork branches and no longer creates PRs.
- Updated `submit_remediation.py` to submit evidence issues only after the human gate and `allow_open_issue=true`. Issue bodies combine Stage 6 issue text, fix solution, patch summary, regression summary, and Stage 7 `reverify.json` evidence.
- Upgraded the remediation config schema to v2.1. Deprecated `fork_url`, `fork_remote_name`, `pr.*`, `allow_push`, and `allow_open_pr`; added `require_evidence_before_issue` with compatibility for the old `require_green_before_pr` alias.

## Safety Rules

- `contract.md` remains the only oracle; fault libraries and wiki pages are advisory only.
- Only `class=sdk_defect` targets with `spec-required` contract authority can enter patch/reverify.
- Non-patchable targets generate evidence issue material only and must not force code changes.
- Stage 6 has no external side effects. Stage 7 submits evidence issues only.

## Verification

- Added `AutoTestFlow/tests/test_stage6_iteration2.py` for patchable subsets, DFX/future profiles, config compatibility, and the issue-only submitter.
- Updated `examples/a2a/remediation_demo` to verify the offline plan → manifest → apply/reverify → dry-run issue submission flow.
