# 07 · LLM Wiki 作为故障库/规格知识源的可行性分析

> 本文回应"知识库/故障库能否以 **LLM Wiki** 形式供 AutoTestFlow 使用"这一问题。
> 在 `05` §b（可插拔文件 vs MCP）和 `06` §五（MCP 作源延后）的基础上，把 **LLM Wiki** 作为第三形态做全谱系分析，重点回答：
> **能否全量接入并准确提取？如何存入？在 AutoTestFlow 哪些环节使用？**
>
> **先说结论**：LLM Wiki **可以采用**，但只能作"由结构化库派生的 NL 建议层"供 LLM 阶段消费，
> 不替代确定性脚本的结构化输入，不作判据（oracle）。

---

## 一、什么是"LLM Wiki"？（通俗解释）

**一句话**：把故障库从"机器读的结构化 JSON"变成"人和 AI 都能轻松读懂的自然语言文章"。

当前的故障库 `rest_api_common_faults.json` 长这样：

```json
{
  "fault_id": "F-REQ-001",
  "name": "body必选参数缺失",
  "test_strategy": {
    "trigger_pattern": "移除请求body中的必选参数字段",
    "expected_behavior": "返回400错误，错误码指示参数缺失",
    "assertion_level": "L2",
    "validation_points": ["HTTP状态码 = 400", "响应body包含error字段"]
  }
}
```

LLM Wiki 把它变成这样的文章：

> **F-REQ-001 · body必选参数缺失**
> 当客户端发请求时漏掉了 body 里某个必填字段，服务端理应拒绝并返回 400 错误。
> 这类问题在 POST/PUT 接口里最常见。历史上 spring-ai-ascend 的 v0.1.0 就踩过这个坑，
> 根因是后端没做入参校验就直接访问了字段……

**三种不同的"变法"**（形态）：

| 形态 | 通俗理解 | 存在哪 | 怎么读 |
|---|---|---|---|
| **A · 仓内文章** | 直接在代码仓里写 Markdown 文章，像写 README 一样 | `Specification_Repository/wiki/F-REQ-001.md` | Agent 按文件名直接读 |
| **B · 语义检索库** | 把文章切碎建索引，像搜索引擎一样按语义找相关 | 向量数据库（FAISS/pgvector 等）| Agent 描述问题，检索返回最相关内容 |
| **C · 外部托管服务** | 把知识托管在专门的问答服务，通过 API 提问 | DeepWiki 类平台或自建文档问答 | Agent 发问，服务端生成答案 |

三种形态共同点：**知识以自然语言表达、面向 AI 语义推理、可由 AI 生成和增量维护**。

## 二、AutoTestFlow 怎么使用故障库？（stage2.6 和 stage6 详解）

在讨论"LLM Wiki 能不能用"之前，先搞清楚 AutoTestFlow 在哪两个关键环节消费故障库、各自怎么用。

### stage2.6 · 故障匹配阶段

**这个阶段做什么**：在拿到被测接口的契约（`contract.md`）之后，AutoTestFlow 会把故障库里的所有故障模式逐一比对，找出"这个接口可能会发生哪些故障"，生成一份**故障注入计划**（`.state/fault_matches.json`）。

**执行方式**：由 `match_faults.py` 这个确定性脚本完成，使用四类规则判断每个故障是否适用于当前接口：

- 接口类型匹配（POST、GET、SSE 流式……）
- 标签匹配（"超时"、"并发"、"参数校验"……）
- 历史缺陷关联（这个接口历史上出过这类 bug）
- 关联资源匹配（涉及某类资源类型）

匹配完成后，脚本还会根据契约的权威性对每个故障的**断言级别做封顶降级**：契约里明确规定的字段才能做硬断言（L2），契约里没规定的只能做观察（L0）。

**可选增强 stage2.6b**：若开启 `--fault-enrich on`，会额外启动一个 LLM 子 Agent（`stage2_6_fault_match.md`），把模糊的验证描述（如"响应 id 字段回带"）精确锚定到契约中具体的条款编号（如 `SPEC-ID-TYPE`），让断言更精准。这个增强步骤是**可选的**，且只修精化细节，不改变确定性骨架。

**这个环节消费故障库的形式**：脚本读取 JSON 结构化字段（fault_id、tags、applicable_scenarios、assertion_level 等），**不需要自然语言叙述**。

### stage6 · 缺陷深度分析阶段（v2.0 自动修复功能）

**这个阶段做什么**：当 stage5 测试报告中出现 `sdk_defect`（有契约背书的真实 SDK 违例）时，stage6 会对每一个真实缺陷分别启动一个 LLM 子 Agent（`stage6_defect_analyze.md`），做侦探式深度分析：

```
读取 contract.md  → 确认这确实是真实违例（有契约背书的 spec-required 字段）
读取执行轨迹      → 找触发请求和违例响应帧（实测证据）
读取源代码        → 定位根因代码（类/方法/文件）
读取故障库        → 拿"同类故障已知规律"作背景知识（规格库知识证据）
     ↓
输出：根因分析 + 代码修复 patch + 回归测试 + GitHub Issue + PR 草稿
```

**这个环节消费故障库的形式**：LLM 子 Agent 读取故障库的**叙事性内容**（为什么会有这个故障模式、典型根因、历史案例、复现步骤），用来丰富 Issue 里"规格库知识背景"那部分证据。这里需要的是"能讲清楚故事"的自然语言，而不是机器读的结构化字段。

### 两个阶段的本质区别

| 对比维度 | stage2.6（故障匹配） | stage6（缺陷深析） |
|---|---|---|
| **目的** | 决定"要测哪些故障" | 讲清楚"为什么这是个 bug" |
| **执行者** | 确定性 Python 脚本 | LLM 子 Agent |
| **对故障库的需求** | 结构化字段（可精确匹配） | 自然语言叙述（可帮助推理） |
| **能吃 LLM Wiki 吗** | **不能**：语义/概率输入会破坏确定性 | **可以**：NL 叙事让推理更准确 |

## 三、全量接入 LLM Wiki：能准确提取吗？如何存入？

### 能否准确提取原始故障库内容

**直接结论**：取决于选哪种形态。

| 形态 | 提取准确率 | 原因说明 |
|---|---|---|
| **A · 仓内文章（按 fault_id 取文件）** | **接近 100%** | 每篇文章对应一个 `fault_id`，文件名即 ID，脚本或 Agent 直接 `Read wiki/F-REQ-001.md`——确定性与读 JSON 完全一致 |
| **B · 语义检索库** | **70–90%**（依模型/索引质量） | 向量检索是概率性的，高相似度不等于语义完全正确；可通过固定 query 模板 + 起点快照降低偏差 |
| **C · 外部托管服务** | **50–85%**（依服务质量） | 问答生成可能出现幻觉、答非所问，且每次调用结果不完全一致 |

**关键判断**：形态 A 能做到与读 JSON 一样精准，同时拥有 NL 的叙事优势。推荐以 A 为基础。

### 如何存入（形态 A 的具体方案）

**目录结构**：每个故障 `fault_id` 对应一篇 Markdown 文章，存入 `Specification_Repository/wiki/`，由脚本从结构化 JSON **单向编译生成**（不手写，不双写，JSON 是唯一真相）：

```
Specification_Repository/
  rest_api_common_faults.json      ← 单一真相（source of truth），不变
  wiki/
    F-REQ-001.md                   ← 由 JSON 自动生成
    F-REQ-002.md
    F-TIMEOUT-001.md
    ...
    category-FC-REQ-001.md         ← 分类汇总文章（可选，供 stage6 读取类别背景）
```

**单篇文章格式模板**（以 `F-REQ-001` 为例）：

```markdown
# F-REQ-001 · body必选参数缺失

> 溯源：rest_api_common_faults.json → FC-REQ-001 → F-REQ-001
> 契约锚点：spec_id = SPEC-REQ-PARAMS | 严重程度：高 | 断言上限：L2

## 故障模式（通俗描述）
当客户端发请求时，漏掉了 body 里某个必填字段。服务端理应拒绝请求并返回 400 错误。
这类故障频繁出现在 POST / PUT / PATCH 接口中。

## 怎么触发
移除请求 body 中的必选参数字段，直接发送到接口即可复现。

## 预期行为
- HTTP 状态码 = 400
- 响应 body 包含 `error` 字段
- `error.code` 明确指示参数缺失
- 响应 `id` 字段回带原始请求 id（幂等/可追溯性要求）

## 常见根因
1. 后端未做入参校验，直接访问字段导致 NullPointerException
2. 使用了可选字段注解（如 `@JsonProperty(required=false)`）但业务逻辑实际依赖该字段
3. 字段名大小写不匹配（如客户端 `userId` vs 服务端 `user_id`）

## 历史案例
（由 `record_faults.py` 从 `history_faults` 中自动追加，每次发现新的 sdk_defect 时更新）

## 关联故障模式
- F-REQ-002 · body参数类型错误（同属 FC-REQ-001 请求体缺陷类）
- F-REQ-003 · body参数边界值超限
```

**四条关键设计原则**：

1. **`fault_id` 即文件名**：`F-REQ-001.md`，Agent 可按 ID 确定性取文件，无需语义检索，提取准确率 ≈ 100%
2. **溯源链完整**：每篇文章顶部标注 `fault_id` + 契约 `spec_id`，防止 LLM 引用时产生幻觉
3. **单向派生**：JSON 是 source of truth，文章由脚本生成，不允许手写或双向同步（防内容漂移）
4. **历史自追加**：`record_faults.py` 发现新的 `history_faults` 时，自动更新对应文章的"历史案例"节

## 四、两类使用场景：哪里适合 LLM Wiki，哪里不行

能不能用 LLM Wiki，关键取决于**谁来消费**这些知识。AutoTestFlow 里有两类性质截然不同的使用者：

**第一类：脚本（不能猜，只认结构化数据）**

`match_faults.py` 和 `record_faults.py` 是纯 Python 脚本，它们靠精确读取 JSON 字段来工作：

- `match_faults.py`：读 `fault_id`、`tags`、`applicable_scenarios`、`assertion_level` 等字段做精确匹配和封顶计算
- `record_faults.py`：用 `(spec_id, field)` 组合做精确去重，把新发现的缺陷写回历史

这两个脚本**完全不理解自然语言**，给它们 NL 文章就如同给计算器输入一段文字——没用。

**第二类：LLM 子 Agent（需要"能讲故事"的知识）**

- `stage2.6b` 增强子 Agent：需要读懂 `validation_point` 的**语义**（"响应 id 字段回带"是什么意思？为什么重要？），才能把它精确绑定到契约条款
- `stage6` 深析子 Agent：需要读故障库里的**叙事背景**（历史上这类 bug 怎么出现的、通常根因在哪、怎么复现），用来写出令人信服的 Issue

LLM 读自然语言文章比读 JSON 字段**理解更深、推理更准**，NL 叙事在这里是真实增益。

**汇总**：

| 消费者 | 阶段 | 需要什么形态的知识 | 能用 LLM Wiki 吗 |
|---|---|---|---|
| `match_faults.py`（脚本） | 2.6 | 结构化 JSON 字段（精确） | **不能**：NL 破坏确定性 |
| `record_faults.py`（脚本） | 5 | `(spec_id, field)` 精确去重 | **不能**：NL 歧义导致漏去重 |
| stage2.6b 增强 Agent | 2.6b（可选） | 验证点的语义含义和契约关联 | **可以**：NL 让绑定更准 |
| stage6 深析 Agent | 6（v2.0） | 故障背景/根因/历史/复现叙事 | **可以**：NL 让 Issue 更有力 |

**结论**：LLM Wiki 只适合第二类（LLM 子 Agent），不适合第一类（确定性脚本）。它**不能替换结构化库**，只能作为**与结构化库并存的 NL 建议层**。

## 五、LLM Wiki 与 AutoTestFlow 设计原则的摩擦点

AutoTestFlow 的核心设计原则（`DESIGN.md` / `shared/rules.md` / `SKILL.md`）对 LLM Wiki 有几处天然摩擦，理解这些才能知道用的边界在哪里：

| AutoTestFlow 的原则 | 与 LLM Wiki 的摩擦 | 怎么化解 |
|---|---|---|
| **确定性可复现**（同样的输入必须产同样的输出） | 语义检索（B/C 态）每次返回结果可能不一样，难以复现和单测 | 仓内文章（A 态）按 `fault_id` 取文件=确定性；B/C 态必须在起点物化快照 |
| **文件即协议**（所有阶段间传递的是文件，不依赖外部服务） | RAG 服务、外部问答服务是"外部依赖"，不是文件 | A 态文章 committed 进仓=仍是文件；B/C 在使用前物化到本地文件 |
| **`contract.md` 是唯一判据**（谁说了算只有契约） | 如果 wiki 文章说"应该返回 XXX"，就成了第二个判据源，有幻觉风险 | wiki 只提供"背景/上下文"，不提供"通过/不通过"的判定；断言仍由 contract 封顶 |
| **禁止洗绿**（不能降低断言来让测试通过） | wiki 不能被拿来作为"故障库说可以这样"的借口放宽断言 | wiki 叙述内容与断言级别无关；断言级别由 contract 权威性决定，不可绕过 |
| **可审计/版本快照** | NL 文章如果随时变化，以前的测试结果就无法复现 | A 态 committed 进 git = 版本化；每次执行时记录使用的 wiki 版本号（同结构化库的 meta.version） |

**一句话归纳**：这些摩擦点**全部只在"wiki 被当成判据或确定性匹配输入"时才会爆发**。只要把 wiki 定位成 advisory（建议/上下文）层，判据始终由 `contract.md` 封顶，这些摩擦就被消除了。

## 六、完整优劣对比（file / MCP / LLM Wiki）

在 `05` §b.4 七维基础上扩展，补充 4 个与 LLM Wiki 强相关的维度：

| 维度 | 可插拔文件（确定性核心） | MCP 外接服务 | LLM Wiki（NL 建议层） |
|---|---|---|---|
| 确定性 / 可复现 | **强** | 弱（依赖服务端状态） | 弱→强：A 仓内文章=强；B/C 起点物化后=中 |
| 可审计 / 版本快照 | **强**（meta 快照库版本） | 需额外版本化 | A git 版本化=强；B/C 需快照+溯源链 |
| 离线可用 | **是** | 否 | A=是；B 自托管可以；C=否 |
| 零知识 · 文件即协议 | **契合** | 偏离（需 Tool/网络） | A=契合（又一个被读文件）；B/C=偏离 |
| 集中化 · 跨项目共享 | 弱（submodule/overlay） | **强** | B/C=强；A=弱 |
| 富查询 / 语义排序 | 无（静态匹配） | 有（结构化查询） | **最强**（NL 语义最贴近 LLM 推理） |
| 运维成本 / 安全 | **低**（无服务，不出域） | 高（托管+数据外发） | A=低；B=中；C=高（同 MCP） |
| **NL 表达力（供 LLM 推理）** | 弱（结构化字段，无叙事） | 中 | **强**（叙事+示例+根因+历史） |
| **作者 / 维护友好** | 中（改 JSON） | 中（改服务端） | **高**：写文章门槛低；可由 LLM 自维护 |
| **幻觉 / 可信风险** | **低**（纯数据） | 低–中 | **高**：需溯源+contract 封顶+不作 oracle |
| **适配消费者** | 确定性脚本（唯一可靠输入） | 脚本（物化后） | **仅 LLM 阶段**（2.6b/6/issue） |

**LLM Wiki 三种形态（A/B/C）子对比**：

| 子维度 | A · 仓内文章 | B · 自建 RAG | C · 外部托管 |
|---|---|---|---|
| 离线/确定性 | **强**（committed，按 fault_id 确定性取） | 弱（语义非确定，可固定 query+快照缓解） | 弱（在线、非确定） |
| 运维/基建 | **无**（纯文件） | 中（向量库+嵌入+检索服务） | 高（托管服务/订阅） |
| 语义检索能力 | 中（靠目录/关键字/按 fault_id） | **强**（语义 top-k） | **强**（NL 问答） |
| 集中化/共享 | 弱 | 中–强 | **强** |
| 数据外发面 | **无** | 视部署（自托管可不出域） | 高（契约/端点发外部） |

> **三形态各有专长**：`file` 占确定性/离线/可审计；`MCP` 占集中化/服务化；`LLM Wiki` 占 NL 表达力/作者友好/语义检索。三者互不替代，可以叠加。

## 七、LLM Wiki 在哪里真正有价值

LLM Wiki 的增量价值**全部落在 LLM 子 Agent 阶段**，且与 v2.0 自动修复功能强协同：

1. **stage2.6b 语义绑定更准**
   自由文本验证点（如"响应 id 字段回带"）配上 NL 解释（为何重要：幂等/可追溯/契约一致），LLM 子 Agent 能更准确地把它绑定到 `SPEC-ID-TYPE` 等具体 specId，检出 `contract_conflict` 更稳。

2. **stage6 深析 + issue 叙事（最大增益）**
   每个故障模式的**复现步骤 / 常见根因 / 关联历史缺陷 / 修复范式**以 NL 文章形式供 stage6 子 Agent 读取，直接强化 v2.0 Issue 的 **(a) 规格库知识**证据（`stage6_defect_analyze.md` 要求两段证据之一），让"根据规格库哪些知识推断这个 bug 成立"更可信、有迹可循。

3. **跨项目知识共享**（B/C 态）
   一处沉淀、多个被测项目复用故障经验——与 MCP 集中化优势同源。

4. **作者友好 + LLM 自维护**
   写 NL 文章比编辑 JSON 门槛低；可让 LLM 把 `record_faults.py` 沉淀的 `history_faults` **自动摘要成 wiki 文章**（含背景/复现/修复建议），降低知识维护成本。

## 八、风险与护栏

| 风险 | 后果 | 护栏 |
|---|---|---|
| **幻觉**（wiki 写出不被契约背书的"期望"） | LLM 把 wiki 错误描述当成断言依据 | 每篇文章**强制溯源**到 `fault_id` + contract specId；断言仍按 `contract.md` 权威性**封顶**；wiki **不进 matcher、不作 oracle** |
| **漂移**（NL 文章与 JSON 不同步） | 测试拿过时的故障描述做推理 | **单一真相**：JSON 是 source of truth，wiki 由其**单向派生**，不允许双向编辑 |
| **不可复现**（语义检索/外部服务） | 同一个测试不同时间结果不同 | 外部/RAG 态在流水线**起点一次性物化快照**到本地文件，下游全程文件化（同 `05` §b.5） |
| **成本/延迟**（检索+大上下文） | 每次执行都要检索，增加耗时 | 优先 A 态按 `fault_id` 确定性取用，避免全量语义检索；必要时使用 prompt 缓存 |
| **数据外发**（C/远端 B 把契约/端点发出域） | 契约内容被发送到外部服务 | 优先 A（仓内不出域）；若用外部服务，限定字段范围、走鉴权，纳入与 MCP 同级的数据外发评估 |
| **自我认证**（wiki 变成第二判据源） | LLM 用 wiki 说"应该通过"来绕过 contract 判据 | 判据唯一来源仍是 `contract.md`（`DESIGN.md` §2）；wiki 只提供"该测什么/为何是 bug"的上下文，不提供"通过与否"的判定 |

---

## 九、推荐架构与演进路线

**整体架构定位（单一真相 + 派生建议层）**：

```
结构化故障库 rest_api_common_faults.json   ← 单一真相（source of truth）
   │
   ├──► match_faults.py / record_faults.py        [确定性脚本：吃结构化 JSON]
   │       ↑ 不变，始终读 JSON
   │
   └──(单向编译派生)──► LLM Wiki（NL 建议层，advisory）
          │  每篇文章顶部溯源到 fault_id + contract specId
          └──► stage2.6b / stage6 / issue          [LLM 子 Agent：吃 NL 文章]
                 判据仍由 contract.md 封顶；wiki 不作 oracle
```

**演进路线（按需推进，越往后越重）**：

- **Phase A（推荐先做，低风险，即时可用）**
  由结构化 JSON 库生成 per-fault / per-category 的 `Specification_Repository/wiki/*.md`，committed 进仓（含 trigger/repro/根因/关联历史/contract 锚点），stage2.6b / stage6 按 `fault_id` 确定性读取相关文章。
  **优点**：无新基建、不出域、契合"文件即协议"，直接增益 v2.0 Issue 叙事，且可作为 B/C 态的文章素材来源。

- **Phase B（按需扩展，规模化语义检索）**
  若故障模式规模扩大（超过 200 条），或需要跨类别语义检索 → 自建 RAG 索引这些文章；查询在流水线**起点物化快照**到本地（同 `05` §b.5），保确定性/可复现。优先自托管以不出域。

- **Phase C（跨团队集中化，可选）**
  若需要跨团队、跨项目共享故障知识 → 外部 DeepWiki/MCP 作"源/注册中心"，仍在起点物化快照、下游文件化。此时 LLM Wiki 与 MCP **叠加**（DeepWiki 经 MCP 暴露）。

**三形态关系总结**：`file = 确定性核心`、`MCP = 集中化源`、`LLM Wiki = NL 建议层`，**三者非互斥、可叠加**——典型终态是"结构化文件喂脚本 + 派生 NL wiki 喂 LLM 阶段 +（可选）经 MCP 集中化分发，起点物化快照"。

**是否可采用（明确结论）**：**可以采用**，须满足四条边界：
① 仅作 **advisory NL 层**供 LLM 阶段消费；
② **单一真相在结构化库**，wiki 单向派生，不双写；
③ **不作 oracle**（判据仍由 `contract.md` 封顶）、**不替代脚本的结构化输入**；
④ B/C 态（外部/检索）必须在起点**物化快照**到本地文件。

落地**推荐 Phase A 起步**：仓内派生文章，与 v2.0 stage6/issue 即时协同，零基建、零数据外发、100% 提取准确率。

---

## 十、一句话结论

LLM Wiki 可作"**由结构化库单向派生、供 LLM 子 Agent 推理的 NL 建议知识层**"被 AutoTestFlow 采用——Phase A 仓内文章方案可做到 100% 准确提取，显著增强 stage2.6b 语义绑定与 stage6/issue 的规格库知识叙事；但**不可替代确定性脚本的结构化输入、不可作 oracle**；集中化/语义检索按需演进并一律**起点物化快照**——与 `file 确定性核心 + MCP 集中化源` 三位一体、互不替代。
