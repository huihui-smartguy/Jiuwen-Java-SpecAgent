# 01 · 故障库材料深度分析

> 对象：`Specification_Repository/`（`README.md` 847 行、`rest_api_common_faults.json` 993 行、`spec.md`）。
> 目的：拆清楚"用户沉淀了什么、用什么结构表达、哪些字段能直接驱动用例生成、与真实流水线有何差距"。

---

## 一、故障库是什么：定位与哲学

`spec.md` 一句话点题：**"该目录是与测试智能体对接的规格库目录。"**
`README.md` 进一步定义它是一个**结构化的缺陷模式知识库**，包含三层：

- **通用故障模式**：适用于所有 REST/JSON-RPC 项目的常见缺陷（51+ 模式）；
- **项目特有故障**：针对特定项目的泛化故障模式（用 `{parameter}` 占位符参数化）；
- **历史缺陷记录**：从测试执行结果中自动积累的真实缺陷（`history_faults`）。

其价值主张（README §一）：覆盖率 70-80% → 90-95%、历史缺陷自动注入 + 行业故障模式增强缺陷发现、
自积累 + 版本化 + 可追溯降低维护成本、全局 + 项目自适应标准化质量。

> 关键判断：这恰好是 AutoTestFlow **当前缺的那一块**——它有"形态正确"的判据（`contract.md`），
> 但没有"应该测哪些失败模式"的经验判据。故障库补的就是后者。

## 二、`rest_api_common_faults.json` 的结构（精确 schema）

### 2.1 顶层结构

```json
{
  "meta":            { "version": "1.1.0", "scope": "global", "description": "...",
                       "last_updated": "2026-01-18", "maintainer": "test_engineering_team" },
  "fault_categories": [ /* 8 个分类对象 */ ],
  "history_faults":   [ /* 5 条历史缺陷 */ ]
}
```

### 2.2 分类对象

```json
{
  "category_id":   "FC-REQ-001",
  "category_name": "请求体缺陷",
  "description":   "请求体相关的常见故障模式",
  "faults":        [ /* 该类下的故障条目 */ ]
}
```

8 大分类、51+ 故障：

| 分类 ID | 名称 | 故障数 | 典型 |
|---|---|---|---|
| FC-REQ-001 | 请求体缺陷 | 11 | F-REQ-001…011 |
| FC-RES-001 | 响应体缺陷 | 8 | F-RES-001…008 |
| FC-PROTO-001 | 协议合规缺陷 | 7 | F-PROTO-001…007 |
| FC-STATE-001 | 状态流程缺陷 | 6 | F-STATE-001…006 |
| FC-SSE-001 | SSE 流式缺陷 | 6 | F-SSE-001…006 |
| FC-SEC-001 | 安全相关缺陷 | 8 | F-SEC-001…008 |
| FC-PERF-001 | 性能相关缺陷 | 4 | F-PERF-001…004 |
| history_faults | 历史缺陷（动态） | 5+ | F-HIST-001…005 |

### 2.3 单条故障条目（核心 schema）

```json
{
  "fault_id":    "F-REQ-001",
  "name":        "body必选参数缺失",
  "description": "请求body中缺少必选参数字段",
  "severity":    "高",                          // 高 / 中 / 低
  "test_strategy": {
    "trigger_pattern":   "移除请求body中的必选参数字段",     // 怎么触发 → 用例输入
    "expected_behavior": "返回400错误，错误码指示参数缺失",   // 正确行为 → 断言目标
    "assertion_level":   "L2",                              // L0 / L1 / L2
    "validation_points": [                                  // 逐条可断言点 → 断言 1:1
      "HTTP状态码 = 400", "响应body包含error字段",
      "error.code指示参数缺失", "响应id字段回带"
    ]
  },
  "applicable_scenarios": ["POST", "PUT", "PATCH"],  // 适用 HTTP 方法 → 端点过滤
  "tags": ["参数校验", "必选字段", "边界测试"]        // 维度标签 → 定位/分组
}
```

### 2.4 历史缺陷条目（比通用故障多 4 个字段）

```json
{
  "fault_id":    "F-HIST-001",
  "source":      "TC-A2A-001执行失败",     // 缺陷来源
  "description": "响应id字段类型不一致（int vs string）",
  "severity":    "高",
  "discovered_date": "2026-01-18",         // 发现日期
  "test_case_id":    "TC-A2A-001",         // 关联用例（→ 历史匹配触发）
  "test_strategy":   { /* 同上 */ },
  "resolution_status": "open",             // open / fixed / closed
  "tags": ["历史缺陷", "类型一致性", "契约验证"]
}
```

> 历史缺陷的 `test_case_id` 是闭环与优先级提升的钩子：它把"哪条用例曾踩过这个坑"沉淀下来，
> 下次直接据此把对应 FP 提升为 P0、并复现该断言。

## 三、故障字段如何驱动用例生成（关键映射）

故障库的设计本质是**把"缺陷经验"参数化成"可执行的测试意图"**。逐字段对应到一条用例的生成：

| 故障字段 | 在用例生成中的角色 | 例（F-REQ-011） |
|---|---|---|
| `fault_id` | 用例溯源标识（新字段 `fault_ref`） | `fault_ref: "F-REQ-011"` |
| `name` / `description` | 用例标题与说明 | "SendMessage-故障验证-关联资源不存在" |
| `severity` | 优先级映射（高→P0/P1） | 高 → P1（历史命中则 P0） |
| `test_strategy.trigger_pattern` | **构造输入/载荷** | `metadata.agentId='nonexistent-{uuid}'` |
| `test_strategy.expected_behavior` | **断言目标（应当怎样）** | "返回错误响应或终态 FAILED" |
| `test_strategy.assertion_level` | 断言严格度初值（会被契约封顶） | L2 |
| `test_strategy.validation_points[]` | **逐条断言（1:1 落地）** | "响应包含 error 或 status=FAILED"… |
| `applicable_scenarios[]` | 端点/方法过滤 | POST/PUT/PATCH |
| `tags[]` | 匹配/分组（映射到 branch 类别） | 参数校验/资源不存在 → exception |

> 即：`trigger_pattern → 输入`、`validation_points → 断言`、`applicable_scenarios + tags → 定位模块`、
> `severity → 优先级`。这套映射是"故障驱动用例"可自动化的根本原因。

## 四、故障库已提出的注入意图与 contract-first 原则

故障库 `README.md` **本身就已经构想了**它该怎样接入 AutoTestFlow（§3.3），这点很重要——
说明用户的设计意图已经存在，本设计是对它的工程化落地与校正：

- **Stage 1（需求分析）**：提升历史缺陷 FP 优先级（`F-HIST-006 → P0`），并在 FP 上记
  `history_fault_refs: ["F-HIST-006"]`。
- **Stage 3a（场景富化）**：把故障注入到场景 `branches.exception`，条目带 `fault_ref` + `trigger` +
  `expected` + `validation_points`（README 给出的样例字段名与本设计一致）。
- **Stage 3b（用例设计）**：生成故障导向用例，`case_id` 带 `fault_ref`，`validation_points` 来自故障策略。
- **Stage 4**：自动积累新发现的缺陷（`new_faults_detected.json`）。

### contract-first 原则（README §5.3，与流水线 `rules.md` 完全吻合）

- 故障的 `expected_behavior` 必须与 `contract.md` 对齐；
- `contract.md` **未定义** → 断言降级为 L0/L1（观察项）；
- `contract.md` **已定义** → 可用 L2；
- **冲突时契约优先**，故障降为观察项。

这与 `AutoTestFlow/shared/rules.md` §2「判据来源纪律（最高红线）：断言只能源自 `contract.md` 或
Grep 核实源码」是**同一条纪律**。故障库没有另起炉灶，而是显式服从契约权威——这让集成在断言安全上没有矛盾。

### 四类匹配触发（README §5.2）

1. **端点匹配**：`applicable_scenarios` 含目标 HTTP 方法；
2. **标签匹配**：`tags` 与场景类型对齐；
3. **历史缺陷匹配**：`test_case_id` 命中已有用例；
4. **关联资源匹配**：检测请求参数中的引用字段（如 `agentId/userId/sessionId/parentId`）。

## 五、材料评估：优点与需对齐的差距

### 5.1 优点（直接可用）

- **schema 规整、字段语义清晰**：`trigger/expected/validation_points/assertion_level` 四件套天然就是
  "输入—预期—断言—强度"，几乎可被确定性脚本直接消费，无需 LLM 兜底即可覆盖高价值故障。
- **已内建版本化与治理**：`meta.version`（1.1.0）+ README §4.4 语义化版本策略；全局 vs 项目 overlay
  （`{parameter}` 占位 + `example_1_a2a_project` 等配置示例）——与 Skill"全局通用 + per-SUT 专化"的
  设计哲学同构。
- **已内建闭环意识**：`history_faults` + `resolution_status` 为"执行→沉淀→复现"闭环留好了数据位。
- **与流水线纪律不冲突**：contract-first、assertion_level（L0–L2）与 `rules.md` 的分级和判据红线一致。

### 5.2 需在实现阶段对齐的差距（不影响本设计成立，但实现时必须处理）

1. **README 引用了尚不存在的脚本/路径**：如 `update_fault_library.py`、`validate_fault_contract.py`、
   `.claude/skills/AutoTestFlow/shared/fault_library/rest_api_common_faults.json`（实际文件在
   `Specification_Repository/`）。→ 实现阶段需以真实产物（`match_faults.py` / `record_faults.py` /
   `.state/fault_matches.json`）对齐 README，避免误导。
2. **"端点匹配"不能按 HTTP 动词直连**：`applicable_scenarios` 是 HTTP 方法（POST/PUT…），但 a2a 是
   **单路由 `POST /a2a`、按 body 内 JSON-RPC `method` 分发**；且流水线的 `entry_catalog` schema
   **不含 HTTP 动词字段**（详见 `02` §"关键事实"）。→ 匹配须经 `contract.md` 的 specId + 标签 +
   历史 + 关联字段联合判定，verb 仅作宽松过滤。
3. **故障爆炸隐患**：51 故障 × N 端点 × N 场景若做笛卡尔积会撑爆用例量。→ 需在匹配脚本里落 README §5.2
   的按类配额上限（详见 `03` §风险）。
4. **断言级别要"被契约封顶"而非"照搬"**：故障条目里的 `assertion_level: L2` 是**期望上限**，实际能否 L2
   取决于 `contract.md` 是否把该字段列为 spec-required。→ 这正是 `match_faults.py` 的契约调和职责。

## 六、本章结论

故障库材料**结构成熟、语义清晰、且与流水线纪律天然兼容**，可以被一个确定性步骤直接消费来生成
故障导向用例；它甚至已经写好了接入设想（Stage 1/3a/3b/4 + contract-first + 四类触发）。
真正要做的，不是"重新发明"，而是：**用一个匹配脚本把它落到 `contract.md` 已就位的那个时点，
按契约封顶断言，再以 `fault_ref` 用例下传**——这正是 `02`（定位注入点）与 `03`（集成设计）要解决的。
