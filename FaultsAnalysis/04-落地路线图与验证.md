## 二、风险与缓解

| 风险 | 缓解 |
|---|---|
| **故障爆炸/组合膨胀**（51 故障 × N 端点 × N 场景） | 匹配器内落 README §5.2 按类配额（≤3 exception/场景、≤2 quality、≤5 安全/项目）；遵守 3b 既有 P1/P2 ≤15 用例/场景与截断序；P0 历史故障豁免。匹配器只输出有界计划，非笛卡尔积 |
| **契约静默时误报** | 调和把静默/冲突点降为 L0/L1 观察；stage4 `sut_unsatisfied`（skip 非失败）吸收其余；只有 spec-required 背书的违例才成 `sdk_defect`。无契约证据 ⇒ 无硬断言 |
| **库漂移/历史缺陷重复** | `record_faults.py` 按 `hash(spec_id+field+rpc_method)` 去重；用例本就带 `fault_ref`（已知故障复现）则跳过；缺省写 overlay/暂存，手工 PR 晋升全局 |
| **RPC SUT 端点 join 错**（`entry_catalog` 无 verb；a2a 按 body `method` 分发） | 匹配器从 `contract.md` + RPC 方法清单建方法清单，verb 宽松匹配，靠 specId/标签/历史/关联字段区分。作为 RPC 处理的规范写法 |
| **性能** | 匹配器是纯 Python 集合/字典操作，输入 ~1k 行 JSON + `contract.md`，亚秒级（同 `select_p0.py`）；v1 热路径无 LLM |
| **向后兼容回归** | 每个钩子以"`.state/fault_matches.json` 存在"为条件；缺库 ⇒ 与今天逐字节相同；纯需求模式天然禁用 |
| **README 期望过时** | Phase 3 把 README 对齐真实产物，避免误导操作者 |

## 三、端到端验证计划（以 a2a 为锚）

目标：证明集成能逼出旧 happy-path 套件漏掉的真实模块缺陷。

1. **匹配器单测（确定性，无需 SUT）**：构造夹具 `contract.md`（含
   `SPEC-RESP-WRAP=result.task`、`SPEC-ID-TYPE`、`SPEC-ENUM=TASK_STATE_*`、`SPEC-SSE`、`SPEC-ERR-32603`）
   + 真实 `rest_api_common_faults.json`；跑 `match_faults.py`；断言 `fault_matches.json` 含
   F-REQ-011（reference_field `metadata.agentId`）、F-HIST-001..005 为 P0、F-SSE-* 落流式；且某无对应
   specId 的点标 `reconciliation:downgraded` 为 L1；`contract_conflict` 降为观察。**这是契约安全证明，
   无需运行 SUT。**
2. **带计划跑 stage3b**：确认新用例带 `fault_ref`、`case_kind ∈ {异常/质量/边界}E2E`、`oracle_refs` 等于
   调和后计划项，且每条仍 ≥1 条 spec-required L2；确认 `merge_test_design.py` 在 `test_design.json` 中保留
   `fault_ref`。
3. **确认覆盖位移**：对比前后 `case_kind` 分布——审计的 ~58% happy-path / 2 负向 / 0 流式故障应转为
   有意义的 exception+quality+SSE 覆盖（按 `fault_ref` 用例计数）。
4. **对真 SUT 跑 stage4**（`POST /a2a`、`GET /.well-known/agent-card.json`）：执行 F-REQ-011 用例——
   若 SUT 对不存在 agentId 返回成功终态，stage4 判 `sdk_defect`（经 `SPEC-RESP-WRAP`/`SPEC-ID-TYPE`
   背书）——一个 happy-path 从不触碰的真实模块缺陷（资源校验）。执行 SSE 用例——非法 `kind` 或缺终态帧
   → 经 `SPEC-SSE` 判 `sdk_defect`。
5. **跑 stage5**：确认"故障库覆盖"小节列出命中故障，且发现的 `sdk_defect` 按 `fault_ref` 分组进确认缺陷清单。
6. **跑 `record_faults.py --write --target overlay`**：确认新 `F-HIST-*` 追加到 overlay，与 F-HIST-006
   去重（同一 agentId 缺陷**不**产生第二条），`resolution_status:open`，`test_strategy` 以被违 specId 为根。
7. **降级回归**：以 `--faults=off`（或移除库文件）整条重跑，与集成前产物 diff——`test_design.json`
   必须完全一致，证明优雅降级。
