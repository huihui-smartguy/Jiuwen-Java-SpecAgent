# FaultsAnalysis · 故障库引入 AutoTestFlow 的分析与设计

> 本目录是对「如何把 `Specification_Repository/` 故障库引入 AutoTestFlow（最新 SpecAgent）」这一问题的
> **深度分析结论 + 集成设计方案**。本轮仅交付分析与设计文档，**不改动** AutoTestFlow 的 Skill/模板/脚本；
> 工程实现作为后续单独一轮（见 `04-落地路线图与验证.md` 的 Phase 1 文件清单）。

---

## 1. 背景与动机

用 AutoTestFlow 以 `AutoTestFlow/examples/a2a` 为锚点测 spring-ai-ascend（A2A/agent-runtime）时，
生成的用例"完整"却**以验证场景是否通畅为前提**——通篇都在确认 happy-path 端到端跑得通，
**针对 a2a 测不出具体模块/缺陷**。审计实证（详见 `02`）：

- 12 条用例中约 **58% 是正向 happy-path**，**仅 2 条真实负向**（parse error / method-not-found）；
- **流式 SSE 故障、超时、并发、资源耗尽、错误响应质量 ≈ 0 覆盖**；
- `testing_recommendations.md` 已明列却**未被测到**的边界：30s 阻塞超时快照、SSE 必须在终态后关闭、
  Runtime 就绪门拒绝、Handler 异常须在 SSE 末尾追加 error 帧。

根因：当前流水线的判据（Oracle）来自"需求 + 真实契约"，它能保证"形态正确、流程跑通"，
但**缺少一份"应该测哪些失败模式"的缺陷经验知识**。用户在 `Specification_Repository/` 沉淀的故障库，
正是这份知识。问题是：**该让 AutoTestFlow 怎么用它？**

## 2. 一句话核心结论

> **把故障库作为"可插拔的外部知识源（规格库）"通过文件注入流水线——而不是把故障知识硬编码进 Skill 提示词/模板。**
> 这不仅是更优的工程选择，更是 AutoTestFlow **架构里早已预留、明确等待接入**的扩展位。

三条硬证据（均为源文件原文）：

1. `AutoTestFlow/DESIGN.md` §3：「…应有一套**业务约束/架构知识的规格库**作为客观评判判据。**当前该规格库缺位**…
   缺规格库 → 人工裁决补位评判标准；**规格库就位后可降级为自动**。」
2. `AutoTestFlow/SKILL.md` §人工裁决：「本 skill 当前**没有业务约束/架构知识的规格库**…规格库就位后，
   这两个门可降级为自动通过。」
3. `Specification_Repository/spec.md`：「该目录是**与测试智能体对接的规格库目录**。」

故障库就是这个 seam 的**首个实例**。再叠加流水线"**文件即协议 + 零知识调度**"的内核——`contract.md`
本身就是一个承载权威、跨阶段下传的**文件**——故障库作为"被确定性步骤消费、再以文件下传"的输入，
与现有架构同构；硬编码反而会成为仓库里唯一违背"文件即协议"的设计。

## 3. 硬编码 vs 可插拔（七维对比摘要）

| 维度 | 硬编码进 Skill 提示词/模板 | **可插拔外部文件（推荐）** | 折中：预编译提示片段 |
|---|---|---|---|
| 耦合 | 紧：改故障要动多个模板 | **松：故障是数据，模板只引用计划文件** | 中：构建步骤耦合 库→片段 |
| 可维护 | 差：51 条故障内联撑爆 6 个模板 | **高：单一 JSON + 语义化版本 + 一个匹配脚本** | 中：多一个易过期的构建物 |
| SUT 可移植 | 破坏"协议无关"不变量 | **保留：全局库 + 项目 overlay** | 保留但片段会随 SUT 过期 |
| 提示膨胀 | 严重（4a/4b 提示已有 <5KB 纪律） | **无：匹配在脚本侧预解析** | 低：片段仍注入提示 |
| 版本化 | 无 | **有：计划文件快照库版本** | 冗余 |
| 可测试 | 难：要跑 LLM 才看得到效果 | **高：匹配脚本可纯单测（仿 select_p0.py）** | 中 |
| 契约安全 | 风险：内联 `L2` 文案诱导越权断言 | **强：脚本先按契约权威性封顶断言级别** | 风险：片段携带原始 L2 进提示 |

结论：**可插拔方案严格占优**。完整论证见 `03-集成设计方案.md` §1。

## 4. 设计要点速览（详见 `03`）

- **不新增 LLM 阶段，新增一个确定性脚本** `match_faults.py`（仿 `select_p0.py`），插在
  **stage2.5 之后、stage3b 之前**（此时 `contract.md` 已就位，可做契约调和）。
- 它产出新文件 `.state/fault_matches.json`（"故障注入计划"）：按"端点/标签/历史/关联资源"四类触发匹配故障，
  并对每条故障的断言级别按 `contract.md` 字段权威性**封顶/降级/冲突时契约优先**。
- **最小模板钩子**：`stage3b` 读计划 → 产出携带新字段 `fault_ref` 的故障导向用例（断言源仍是 `contract.md`）；
  `stage4` 几乎不变（`fault_ref` 透传，`merge_test_design.py` 已逐字透传整 dict）；`stage5` 增"故障库覆盖"小节。
  **所有钩子以"若计划文件存在"为条件 → 缺库即与现状字节级一致（优雅降级）。**
- **闭环反馈**：新增 `record_faults.py`，把 stage4 中 `class==sdk_defect` 的契约级真实缺陷蒸馏回
  `history_faults`（默认 dry-run、去重、写项目 overlay 不污染全局）。
