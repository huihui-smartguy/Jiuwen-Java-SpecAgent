## 评审 a · README §5.2 引用的"尚不存在脚本"：它们做什么？本 Skill 是否需要？

### a.1 两个缺失脚本的职责（README 原文）

| 脚本 | README 位置 | 输入 | 输出 | 触发阶段 | 职责（README 描述） |
|---|---|---|---|---|---|
| `update_fault_library.py` | 524–527 | `--results-dir .state/results`（含 `.state/results/new_faults_detected.json`）、`--fault-lib …rest_api_common_faults.json` | 写入库的 `history_faults` 字段 | Stage 4 之后 | "测试执行后…从失败结果中提取新缺陷…自动写入故障库的 history_faults" —— **闭环自积累** |
| `validate_fault_contract.py` | 656–659 | `--fault-lib fault_library/project_faults.json`、`--contract .state/contract.md` | `--output fault_library/fault_contract_alignment.md` | Stage 2.5（隐含） | "故障库的 `expected_behavior` 需与 `contract.md` 校准…契约未定义则降级 L0/L1…生成校准报告" —— **契约优先调和 + 对齐报告** |

另有**路径错位**：README 多处（55/445/524）假设全局库在 `.claude/skills/AutoTestFlow/shared/fault_library/rest_api_common_faults.json`，但仓库里实际只存在 `Specification_Repository/rest_api_common_faults.json`，且 `AutoTestFlow/shared/fault_library/` 目录不存在。

### a.2 分析：这两个脚本"该不该建"

把它们的**能力**与上一轮设计（`03-集成设计方案.md`）逐一对位：

- **`update_fault_library.py` ≡ 本设计的 `record_faults.py`**（03 §2.5），且本设计更安全：
  - README 写的是"**从失败结果**提取新缺陷"——这是**危险的**。按 `shared/rules.md` §4 / `DESIGN.md` §4 的 5 分类，"失败"包含 `harness_defect`（脚本自身错）、`sut_unsatisfied`（非协议必需、不算缺陷）、`env_issue`（环境）。若一律入库，会把"非缺陷"沉淀成"历史缺陷"，污染知识、放大误报。
  - 正确做法：**仅收 `class==sdk_defect`**（contract 背书的真实违例），并叠加去重（`hash(spec_id+field+rpc_method)`）、默认 dry-run、默认写**项目 overlay** 而非全局库。
- **`validate_fault_contract.py` 的"调和逻辑"必须有，但不必独立成脚本**：它就是 `match_faults.py` 内的 **reconciliation 步**（03 §2.2/§2.6）——按 `contract.md` 的「字段权威性分级表」给每条故障 `oracle_ref.assert_level` **封顶**：`spec-required`→可 L2、`deployment-config-dependent`/契约静默→降 L0/L1 观察、冲突→契约优先降为观察项。把它做成独立脚本会与 `match_faults.py` 重复读取/重复匹配。
  - 唯一值得保留的是它的**副产物**：一份 `fault_contract_alignment.md` 对齐报告（可观测性）。可让 `match_faults.py` 顺带产出，或并入 stage5 §6"契约权威性观察"。

### a.3 结论与映射表

| README 脚本 | 本设计对应物 | 是否需要"按原名新建" | 落点 | 关键增强 |
|---|---|---|---|---|
| `update_fault_library.py` | `record_faults.py` | **否** | 03 §2.5（stage5 子步） | 仅 `sdk_defect`、去重、dry-run、写 overlay |
| `validate_fault_contract.py` | `match_faults.py` 的 reconciliation 步 | **否**（逻辑内置） | 03 §2.2/§2.6 | 断言级别按契约权威性封顶；可选产出对齐报告 |
| 路径 `.../shared/fault_library/…` | `--fault-lib` 默认探测 `Specification_Repository/` | —— | 03 §2.1 发现约定 | 统一发现约定，避免路径漂移 |

> **一句话**：README 描述的是"愿景脚本"，其能力已被本设计的 `match_faults.py` + `record_faults.py` 覆盖且做了安全收敛。实现轮**不新建这两个名字**，而是**对齐 README**到真实产物，并把"从失败结果提取"明确收敛为"仅 sdk_defect"。

## 评审 b · "故障爆炸"能否用 MCP 外接规格库？交互如何？与可插拔文件孰优？

### b.1 先厘清现状（决定 MCP 是"融入"还是"偏离"）

- AutoTestFlow **全程不使用任何 MCP/外部服务**（对 `AutoTestFlow/` 全树 Grep `mcp`/外部服务：0 命中）。设计/生成阶段的知识仅来自：需求文档、SUT 源码静态分析(stage2)、**对真实 SUT 的探活**(stage2.5 `probe_contract.py`)，以及本设计引入的故障库**文件**。
- 子 Agent 是 **file-in / file-out 的纯文本生成，不发起 Tool 调用**；编排器"零知识调度、文件即协议"（`SKILL.md` 174–195）。
- 因此，**引入 MCP = 给某阶段子 Agent 注入 Tool 调用**，这是对当前执行模型的**结构性偏离**（不是顺势扩展）。

### b.2 MCP 交互模型（如果要做，长什么样）

故障库 MCP server 暴露查询/记录类工具，Agent 以 `contract.md`/`code_facts` 派生的端点与标签为入参，**服务端完成匹配/排序/配额**后只回**有界子集**：

```
# 调用时机：stage2.6（或 3a/3b 子 Agent）
query_faults(
  http_method="POST", rpc_method="SendStreamingMessage", path="/a2a",
  tags=["SSE","流式"], scene_type="quality", limit=5
) -> [ {fault_id, name, severity, trigger, validation_points, assertion_level, tags}, ... ]   # 仅 3–5 条

get_fault(fault_id="F-SSE-001")            -> 单条详情
list_history(test_case_ids=["TC-A2A-002"]) -> 命中历史缺陷（驱动 P0 提升）
record_fault(defect={spec_id, field, rpc_method, expected, actual, case_id})  -> 闭环写回
```

即：Agent 不再"加载整库再筛"，而是"按需问、服务端答"。

### b.3 关键澄清：故障爆炸的根因不在"数据源"，而在"用例组合"

"故障爆炸"指的是**用例数量**的组合膨胀（故障 × 端点 × 场景），它由**配额/选择策略**决定，**与故障数据来自 file 还是 MCP 无关**：

- MCP 的 `limit`/服务端排序，减少的是 Agent **上下文里看到的故障条数**（载入量），这对省 token 有益；
- 但**最终生成多少条用例**仍取决于配额——`match_faults.py` 已按 README §5.2 落了按类配额（≤3 exception/场景 等）。
- 结论：**file 与 MCP 都能解决故障爆炸**；解决它的是"配额"，不是"换数据源"。**故障爆炸不构成采用 MCP 的理由**。

### b.4 七维优劣对比

| 维度 | 可插拔外部文件（推荐核心） | MCP 外接服务 |
|---|---|---|
| 确定性 / 可复现 | **强**：同输入同输出，可单测（仿 `select_p0.py`） | 弱：依赖服务端当时状态/网络，难复现 |
| 可审计 / 版本快照 | **强**：`fault_matches.json.meta` 快照库版本 | 需额外把查询结果版本化才能审计 |
| 离线可用 | **是**：纯本地文件 | 否：需服务在线可达 |
| 零知识 · 文件即协议 | **契合**：又一个被脚本消费、以文件下传的输入 | 偏离：需子 Agent 持有 Tool 权限并处理调用/错误 |
| 集中化 · 跨项目共享 | 弱：靠 git submodule / 复制 / overlay | **强**：一处更新、多方共享 |
| 动态排序 / 服务端筛选 | 无（静态配额，确定但不智能） | **有**：可做 ML 排序、富查询 |
| 运维成本 / 安全 | **低**：无额外服务，数据不出域 | 高：需托管；把 SUT 端点/契约发往外部，引入鉴权与数据外发面 |

### b.5 结论与推荐

- **以 file 为确定性核心**：契合"文件即协议/零知识/可复现/可单测/离线"，且与"换 SUT 只换 overlay"的设计哲学一致。
- **MCP 更适合做"集中式、跨团队、持续演进的故障知识服务"**，但与本流水线的确定性/可复现/零知识相冲突，不宜作为每阶段的**实时**依赖。
- **推荐混合**：把 MCP 当作**源 / 注册中心**，在流水线**起点一次性物化版本化快照**到本地文件，下游全程文件化、可复现：

```
match_faults.py --fault-source mcp://fault-registry/...   # 起点：拉取版本化快照 -> 落地为本地 JSON
   -> 之后 stage2.6/3b/4/5 全部只读本地文件（与现设计一致，零改动）
record_faults.py --write --target overlay [--push-registry mcp://...]  # 闭环：本地 overlay 为主，可选回写注册中心
```

如此既拿到 MCP 的"集中化/可演进"，又保住流水线的"确定/可复现/可审计"。**当前阶段无需 MCP**：先把 file 核心做扎实，MCP 作为后续可选演进（路线图 Phase 3 之后）。

## 评审 c · §3.3"故障库可补缺口"是不是最终给人的"缺陷建议"？依据是什么？

### c.1 结论：不是同一回事，是同一因果链路的两端

| | `02` §3.3 "可补缺口" | 最终"缺陷建议"（给人看的结论） |
|---|---|---|
| 性质 | **输入 / 动机 / 风险假设** | **输出 / 经验证据** |
| 回答的问题 | "**应该补测什么**"（哪些失败模式没测） | "**实测发现了什么真问题**" |
| 生成依据 | 当前 `test_design.json` 覆盖审计（~58% happy-path、SSE 故障 0…）× 故障库 8 类 taxonomy 的**对照** | stage4 执行结果 `.state/results/*.json` 中 `class==sdk_defect`（断言基于 `spec-required` specId、写法正确、SUT 偏离契约），**contract 背书** |
| 落地位置 | `02` §3.3 表 / `match_faults.json` | `report.md` **第4部分「确认缺陷清单（仅 sdk_defect）」**（5 列：用例｜违背 specId｜contract 期望｜SUT 实际｜代码位置/修复建议）；另含第7部分静态 CD / 需求-实现 GAP |
| 产出时机 | **设计前**（选什么去测） | **执行后**（测完得到什么） |

### c.2 因果链路（缺口如何"变成"缺陷建议）

```
覆盖缺口(§3.3)            ——风险假设：哪些失败模式没覆盖
   │  match_faults.py 选故障（四类匹配 + 契约封顶）
   ▼
故障注入计划 fault_matches.json
   │  stage3b 产出带 fault_ref 的故障用例
   ▼
test_design.json（故障导向用例）
   │  stage4 执行 + 5 分类
   ▼
.state/results/*.json  ──分流──> pass / harness_defect / sut_unsatisfied / sdk_defect / env_issue
   │                                                              │ 仅取 sdk_defect
   ▼                                                              ▼
stage5 report.md 第4部分「确认缺陷清单」 = 给人的缺陷建议（含"代码位置/修复建议"）
```

### c.3 必须澄清：**填补缺口 ≠ 发现缺陷**

- 缺口是"问对问题的集合"，把它补上只是**问了更多对的问题**；
- 多数故障用例执行后会 **PASS**——这说明 SUT **正确处理了**该失败模式，是**保障/信心**，不是缺陷；
- 只有"用例可跑通 + 断言基于 spec-required + SUT 偏离契约"才落 `sdk_defect`，才进缺陷建议；
- `sut_unsatisfied`（非协议必需的期望未达成）只进 `report.md` **第5部分**"SUT 未满足项（已忽略）"，**不计缺陷、不洗绿、不改断言**（`rules.md` §4 红线）。
- 因此故障库的价值是**扩大缺陷暴露面（可能性）**，而非**直接产出缺陷**。它提升的是"该测的都测了"的**置信度**与"漏测导致的假阴性"的下降，最终是否成为缺陷建议，由 stage4 的契约背书判定。

### c.4 "缺陷建议"在 stage5 的确切构成

智能体最终向人传达的，不止动态缺陷一项（`stage5_report.md` 七部分）：
- **第4部分 确认缺陷清单（仅 sdk_defect）**：动态执行发现、contract 背书 —— 故障库主要增益此处；
- **第7.1 静态代码缺陷（CD）**：stage2 代码扫描得到（`stage_summary.json::cd_list`）；
- **第7.2 需求-实现 GAP**：stage2 gap 分析得到（`gap_list`）；
- **第5/6 部分**：`sut_unsatisfied` 观察项、契约权威性观察 —— **不是缺陷**，是给人参考的观察。

> 一句话：§3.3 是"**测什么**"的依据（风险驱动的覆盖缺口），缺陷建议是"**测出了什么**"的结论（契约背书的 sdk_defect）。两者经 match→设计→执行→5分类 的链路相连，但**不能划等号**。

## 5 · 小结

- **a**：两个 README 脚本**不必按原名新建**——能力已由 `match_faults.py`（含契约调和）+ `record_faults.py`（闭环，仅 sdk_defect/去重/overlay/dry-run）覆盖；实现轮**对齐 README、统一发现约定**即可。
- **b**：**故障爆炸由配额解决，与数据源无关**；file 为确定性核心，MCP 适合做集中式知识服务但偏离零知识/可复现——**推荐"file 核心 + MCP 作可选源（起点物化快照）"的混合**，当前阶段无需 MCP。
- **c**：§3.3 缺口是"**该测什么**"的输入，缺陷建议是"**测出什么**"的输出（stage5 §4 的 `sdk_defect`，contract 背书）；**填补缺口 ≠ 发现缺陷**。

**对实现轮的影响**：主线（可插拔文件为核心）不变；新增三条增强——①对齐 README 到真实产物并收敛闭环安全（仅 sdk_defect）；②预留 `--fault-source` 以便将来 MCP 作源；③修 `sdk_bug`/`sdk_defect` 字段名并澄清 `01/02` 的输入/输出措辞。
