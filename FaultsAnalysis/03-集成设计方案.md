# 03 · 集成设计方案（核心）

> 回答用户的核心问题：**AutoTestFlow 怎么引入故障库——Skill 里直接硬编码，还是更完美的方式？**
> 本章给出推荐结论、完整集成架构、新产物 schema、最小模板钩子、契约安全保障、闭环反馈与治理，
> 并用两条 a2a 实例展示"用例从验流程变为探模块"。

---

## 1. 推荐结论：可插拔外部规格库（不硬编码）

### 1.1 为什么可插拔严格占优

整条流水线建立在 **"文件即协议 + 零知识调度"** 之上：编排器只看文件存在性、每阶段隔离、数据只走文件，
`contract.md` 就是典范——一个**文件**把权威从 stage2.5 带到 stage3b/4。故障库做成"被确定性步骤消费、
再以文件下传"的输入，与现有架构同构；**硬编码进提示词会成为仓库里唯一违背文件即协议的设计**。

三条架构事实使"外置"成为唯一一致的选择：

1. **SUT 无关是硬不变量**：`SKILL.md`/`rules.md`/`java_scan_guide.md`/`examples/a2a/README.md` 反复强调
   "A2A 仅为锚点，非硬编码"。故障库的 `meta.scope:"global"` + 项目 overlay 正好镜像"通用内核 + 专化外挂"。
   把 51 条 REST/JSON-RPC 故障烤进提示词，等于把协议假设硬塞进一个刻意保持协议无关的 skill。
2. **规格库 seam 已写明**：`rules.md` §3 要求把人工裁决结论"沉淀进规格库（`.state/spec_decisions.json`
   或文档）"，`DESIGN.md` §3 / `SKILL.md` 许诺"规格库就位后可降级为自动"。故障库是该 seam 首个实例，
   走文件能让 seam 复用于未来其它规格类型（业务约束、架构规则）。
3. **可维护/版本化/闭环要求数据文件**：库已有 `meta.version`、语义化版本策略、`history_faults` 自积累。
   "把发现的缺陷自动追加进 JSON"是平凡操作且本身就是文件即协议；"把缺陷追加进提示词"不成立。
   库与 skill 演进节律不同，**必须独立版本化**。

### 1.2 七维对比

| 维度 | 硬编码进提示词/模板 | **可插拔外部文件（推荐）** | 折中：预编译提示片段 |
|---|---|---|---|
| 耦合 | 紧：改故障要跨阶段改模板 | **松：故障是数据，模板引用计划文件** | 中：构建步骤耦合 库→片段 |
| 可维护 | 差：51 故障内联撑爆 6 模板 | **高：单 JSON + 语义版本 + 一脚本** | 中：多一易过期构建物 |
| SUT 可移植 | 破坏 SUT 无关不变量 | **保留：全局库 + 项目 overlay** | 保留但片段随 SUT 过期 |
| 提示膨胀 | 严重（4a/4b <5KB 纪律） | **无：脚本侧预解析** | 低：片段仍入提示 |
| 版本化 | 无 | **有：计划文件快照库版本** | 冗余 |
| 可测试 | 难：要跑 LLM 才见效 | **高：脚本可纯单测（仿 select_p0.py）** | 中 |
| 契约安全 | 风险：内联 L2 文案诱导越权 | **强：脚本先按契约权威性封顶** | 风险：片段携原始 L2 入提示 |

### 1.3 折中方案为何被支配

"用确定性构建步骤把库编译成缓存提示片段、注入 stage3b"是可行但被严格支配：它仍向提示注入内容
（重新引入膨胀、违背 4a/4b <5KB 纪律），仍需同样的匹配逻辑，还多一个会过期的构建物。
推荐方案**已经有一个确定性步骤**——把它的产物做成**计划文件**（而非提示片段）纯属更优：
stage3b 像读 `contract.md`、`s3a_enriched_index.json` 一样把它当协议来读。

## 2. 集成架构

### 2.1 发现/配置约定 + 优雅降级

编排器保持零知识，只多检查"故障库文件是否存在"。

- **新增可选参数**（`SKILL.md` 参数表）：
  - `--fault-lib`：全局库路径，默认自动探测 `Specification_Repository/rest_api_common_faults.json`；
  - `--fault-overlay`：项目级 overlay，默认自动探测 `{project_root}/fault_library/project_faults.json`；
  - `--faults=auto|on|off`：默认 `auto`（探测到库文件即启用）。
- **优雅降级（硬要求：缺库即与现状字节级一致）**：
  - 未解析到库 / `--faults=off` → **跳过 2.6 步骤，绝不生成 `.state/fault_matches.json`**；
  - stage3b/stage5 的新逻辑**严格以"该文件存在"为条件**（`if exists(...) then … else 现状`），
    文件不在 → 行为与今天逐字节相同；
  - **纯需求模式天然禁用**故障匹配：无 `contract.md` 可调和，编排器根本不跑 2.6，stage1/3b 两道人工门
    原样保留。

### 2.2 新增确定性步骤 2.6：`scripts/match_faults.py`（故障匹配器）

**放置（论证）：新增 2.6，紧接 stage2.5 之后、stage3a 之前。** 在 2.5 之后是因为契约调和需要
`contract.md`；在 3b 之前是因为 3b 要消费计划；紧贴 2.5 是为保持它是干净的确定性门（与
`probe_contract.py`/`select_p0.py` 的节律一致）。

**形态：脚本优先，LLM 增强可选；v1 纯脚本。** 理由：仓库既有"join/选择/调和=纯 Python"的范式
（`select_p0.py` 等）。故障匹配就是集合交（`applicable_scenarios`/`tags`）、等值 join（`test_case_id`）、
查表降级（故障→specId→断言封顶）——无需 LLM，纯脚本可单测、可复现（验证计划依赖此）。
LLM 仅在**模糊处**有用：(a) 当 `validation_points` 自由文本→specId 不是表驱动时的绑定；(b) 项目 overlay
`{placeholder}` 故障的具体取值参数化。这部分放进**可选的 stage2.6 LLM 增强子 Agent**，仅 enrich 脚本
标了 `needs-llm-bind` 的字段；MVP 先不做（表驱动已覆盖 id 类型/响应包装/枚举/SSE/错误码/资源不存在等
高价值故障，恰是 a2a 历史缺陷）。

**匹配逻辑（README §5.2 四类触发，适配真实端点模型）：**

1. **端点/方法匹配**：从 `contract.md`（含 `SPEC-ERR-*`/`SPEC-SSE`…）+ RPC 方法清单构建 SUT 的
   `{http_method, rpc_method, path}` 清单（**不**依赖 `entry_catalog` 的 verb——它没有 verb 字段）。
   a2a 下所有调用都是 POST，verb 匹配宽松，真正区分靠下面三项。
2. **标签匹配**：`tags` ∩ 场景/分支语义，按 README §5.2 映射到 branch 类别（FC-REQ→exception、
   FC-RES→quality、FC-PROTO→exception、FC-STATE→boundary、FC-SSE→quality、FC-SEC→专门安全、
   FC-PERF→quality/dfx）。
3. **历史匹配**：`history_faults[].test_case_id`（如 `TC-A2A-001`）+ overlay `related_endpoint` join 到
   FP/场景 → 这些**提升为 P0**。
4. **关联资源匹配**：检测 entry `params` 里的引用字段（`metadata.agentId`、`params.id`…）触发
   F-REQ-011 / F-STATE-002——正是 a2a 的 F-HIST-006。

**contract-first 调和（安全内核）：** 对每条命中故障，脚本把其 `validation_points` 与 `contract.md`
字段权威性分级表求交，计算 `resolved_assert_level`：

- 命中 `spec-required` specId → 保留至多 **L2**（`rules.md` 的合格线）；
- 命中 `deployment-config-dependent`，或 `contract.md` 静默 / `needs-runtime-verify` →
  **降为 L0/L1（观察项）**，记 `reconciliation:"downgraded"` + 原因；
- 故障 `expected_behavior` 与 `contract.md` **直接冲突** → **契约优先**，故障降为观察项，
  记 `reconciliation:"contract_conflict"`。
  → 保证**故障断言永不超过契约权威，且在 stage3b 看到之前就已封顶**。

字段命名对齐既有 schema（`fault_id`/`assert_level`/`spec_id`/`authority`/`oracle_refs`/`priority`/
`branch_class`/`source_scene`）：

### 2.4 最小模板钩子（向后兼容，均以"计划文件存在"为条件）

1. **`SKILL.md`（编排器）**：参数表加 `--fault-lib/--fault-overlay/--faults`；阶段映射表插一行
   "2.6 故障匹配 | (无模板，纯脚本) | `scripts/match_faults.py` | `.state/fault_matches.json` | — | ❌"
   （置于 2.5 与 3a 之间）；执行流程在 2.5 后加："若 `--fault-lib` 解析到文件 → 前景执行
   `python scripts/match_faults.py --output-dir {out} --fault-lib {lib} [--fault-overlay {ovl}]`；否则跳过。"
   零知识不变（事后只检查产物存在性）。

2. **`templates/stage3b_batch_design.md`（唯一实质钩子）**：加一条件步骤"读取本批场景的故障匹配"——
   若 `.state/fault_matches.json` 存在，对每个 `source_scene` 读其 `fault_matches`，**额外产出故障导向
   用例**（每命中故障 1 条，遵守 README §5.2 配额，如 ≤3 exception/场景），用例：
   - `case_kind` 由 `branch_class` 决定（exception→异常E2E、quality→质量E2E、boundary→边界E2E）；
   - **新字段 `fault_ref:"F-REQ-011"`**（附加字段，`merge_test_design.py` 逐字透传，安全）；
   - `oracle_refs` **直接取调和后的 `oracle_refs`**（validation_points ∩ contract），断言源仍是
     `contract.md` 的 specId，**完全符合 `rules.md` §2 与 3b 既有 oracle 规则**；
   - `priority` 取自匹配（历史→P0，否则按 3b 既有优先级表）。
   - 既有规则"每条用例 ≥1 条落在 spec-required 的 L2"仍生效：若某故障整体降为观察，用例仍须在底层
     happy-path 字段（如 `id` 回带）带 1 条 spec-required L2——匹配器通过总是带上 `SPEC-ID-TYPE`/
     `SPEC-RESP-WRAP`（当存在）来保证。

3. **`templates/stage4a_p0_verify.md` / `stage4b_batch_gen.md`**：**基本不变**。它们已通用消费 `oracle_refs`
   并按 authority 施加强/弱断言；带 `fault_ref` 的用例只是又一条带 `oracle_refs` 的用例，5 分类已能区分
   `sdk_defect` 与 `sut_unsatisfied`。唯一可选微调：把 `fault_ref` 透传进 `.state/results/{id}.json`
   以便溯源（多一字段，无害）。

4. **`templates/stage5_report.md`**：加一节"故障库覆盖（fault coverage）"——矩阵：命中故障 × 用例 ×
   结果分类；并把带 `fault_ref` 的 `sdk_defect` 单列"故障逼出的缺陷"（复用既有第 4 部分缺陷清单数据，
   按 `fault_ref` 分组）。

5. **`templates/stage1_req_analyze.md`（可选）**：历史缺陷 P0 提升——若已有 history 信息，把命中的 FP 提为
   P0 并记 `history_fault_refs`。注意 stage1 在匹配器之前运行，v1 更干净的做法是**把 P0 提升放进匹配器**
   （它可读 `s1_index.json`），让 3b 直接采纳；stage1 钩子仅当你想让提升体现在 `requirement_analysis.md`
   时才需要。无论哪种都是附加字段（`history_fault_refs`），与 README §3.3 一致。

### 2.5 闭环反馈：`sdk_defect` → 新 `history_faults`

**新增 `scripts/record_faults.py`，作为 stage5 子步（确定性，在 `aggregate_results.py` 之后）：**

- **输入**：`.state/results/*.json`（含 `status`/`class`/`sdk_defect{spec_id,expected,actual}`，加 §2.4.4
  微调后还有 `fault_ref`/`oracle_refs`）+ 对应用例 + 当前库文件。
- **筛选**：仅 `class=="sdk_defect"`（契约背书的真实违例；**绝不**收 `sut_unsatisfied` / `harness_defect`
  ——这正是 `DESIGN.md` §4 / `rules.md` §4 守护的边界）。
- **蒸馏**为 `history_faults` 条目（字段对齐既有 schema：`fault_id`/`source`/`description`/`severity`/
  `discovered_date`/`test_case_id`/`test_strategy{...}`/`resolution_status:"open"`/`tags`）。
  `expected_behavior`/`validation_points` 取自被违 specId（保持契约背书），`test_case_id`=用例 id，
  `source="{case_id} 执行 sdk_defect"`。
- **去重（防库漂移）**：稳定签名 `hash(spec_id + 归一 field + rpc_method)`；若全局/历史已有该签名，
  或用例本就带 `fault_ref`（说明是**已知故障复现**），则**不新增**，仅按需更新 `resolution_status` 观察。
  新 ID 按 README §4.1 取 `F-HIST-{seq}`。
- **写入策略/治理**：默认写**项目 overlay/暂存**（`fault_library/discovered_faults.json` 或
  `project_faults.json` 的 `history_faults`），**不写** `scope:"global"` 全局库（保持全局精选）；
  global ← overlay 的晋升走手工 PR（README §10）。提供 `--target {overlay|global}`；并以 `--write`
  显式开关闸门，缺省**dry-run**（仅打印拟新增条目供 stage5 报告）。

> 闭环全程文件即协议：results 文件 → 脚本 → 库文件，不改任何提示词。

### 2.6 契约安全（汇总四重保障）

1. **暴露前封顶**：`match_faults.py` 在 stage3b 看到之前就按 `contract.md` authority 封顶每个
   `oracle_ref.assert_level`；3b 既有规则本就禁止对 config-dependent/needs-runtime-verify specId 做强断言。
2. **冲突契约优先**：调和标 `contract_conflict` → 故障降为观察，进 stage5 记"需求-实现形态差异"观察
   （与 `SKILL.md` 强制覆盖规则末行一致）。
3. **静默即观察**：契约无对应 specId 的点只能 L0/L1；用例仍须带 1 条 spec-required L2，否则降为纯观察用例。
4. **5 分类终极兜底**：即便故障断言写错，stage4 也只把它路由到 `sut_unsatisfied`（skip 非失败），
   除非是契约背书的 `sdk_defect`。**没有故障能"洗绿"或被高估为缺陷**（`rules.md` §4 红线）。

### 2.7 schema 与治理

- **版本化**：`fault_matches.json.meta.*_version` 快照输入，任一次运行可复现，stage5 可报告所用库版本
  （README §6.1 已预期）。
- **全局 vs 项目**：全局 `rest_api_common_faults.json`（`scope:"global"`，精选）+ 项目
  `project_faults.json`（`extends`、`{parameter}` 占位、`fault_overrides`、`disabled_faults`）——即 README
  §2.2 模型。匹配器加载全局后套用 overlay：`disabled_faults` 过滤、`fault_overrides` 替换 `test_strategy`、
  `project_specific_faults` 追加、`{placeholder}` 用 `configuration_examples`/`parameter_config` 替换。
- **per-SUT overlay**：`--fault-overlay` 指向该 SUT 的 `project_faults.json`；缺省即仅全局。这是
  "用 `examples/a2a/` 专化通用 `reference/`"的故障侧对应物。

## 3. 用例形态变化实例（a2a：从验流程 → 探模块）

### 3.1 F-REQ-011「引用不存在的关联资源」于 SendMessage（即 F-HIST-006 缺陷）

```json
{
  "case_id": "TC_041",
  "name": "SendMessage-故障验证-关联资源不存在(agentId)",
  "test_type": "scenario",
  "dimension": null,
  "case_kind": "异常E2E",
  "priority": "P1",
  "source_scene": "FS-001",
  "fault_ref": "F-REQ-011",
  "preconditions": ["SUT 就绪", "使用一个确定不存在的 agentId"],
  "steps": "1. 构造 SendMessage 请求，metadata.agentId 设为 'nonexistent-<uuid>'\n2. 通过 POST /a2a 发送(Accept: application/json)\n3. 接收并解包响应，检查终态与错误信号\n4. 复查：不产生有效任务副作用",
  "expected": "【输出】result.task.status.state 归一化为 FAILED，或返回 JSON-RPC error；id 回带等于请求 id | 【过程】不出现成功终态 COMPLETED，未创建有效任务",
  "oracle_refs": [
    {"spec_id": "SPEC-RESP-WRAP", "assert_level": "L2", "field": "result.task.status.state", "authority": "spec-required"},
    {"spec_id": "SPEC-ID-TYPE",   "assert_level": "L2", "field": "id", "authority": "spec-required"},
    {"spec_id": "SPEC-ERR-32603", "assert_level": "L1", "field": "error.code", "authority": "spec-required"}
  ],
  "param_overrides": [
    {"param": "metadata.agentId", "values": ["nonexistent-<uuid>"], "step_ref": 1}
  ]
}
```

调和说明：故障原始点"错误码/message 指示资源不存在"在 `contract.md` 无专属 specId（a2a 把通用失败映射到
`-32603` Internal Error），故该点保持 **L1**（对 `error.code` 的观察），而 **L2** 强断言落在
`SPEC-RESP-WRAP`（`result.task.status.state` 经 `normalize_state`）与 `SPEC-ID-TYPE`（`id_eq`）。

agent-runtime 的"资源校验"模块，而非流程是否通畅。**
