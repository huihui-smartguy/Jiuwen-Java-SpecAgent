# 共享规则与跨阶段纪律

> 本文件定义本 skill 所有阶段必须遵守的共用纪律，适用于**任意对外暴露 HTTP/SSE 契约的 Java/Spring SUT**。
> 核心立场：**判据（Oracle）权威性第一**。断言只能基于已校准的契约或核实过的源码，绝不臆测响应形态。

---

## 核心保证

1. **判据先于断言** —— 任何期望值出现前，必须能指回 `contract.md`（stage2.5 校准产物）或 Grep 核实的源码行。
2. **单用例责任制** —— 1 用例 = 1 Agent，独立完成生成→执行→修复→记录，**不允许任何理由合并**。
3. **交互轨迹强制采集** —— 记录器已内置于 `reference/http_client.py`，每条用例必须产出可读请求/响应/SSE 轨迹。
4. **就绪门后才执行** —— stage4 执行前必须探活 SUT，不就绪即 `env_issue` 并保留代码，绝不“洗绿”。
5. **断言必须能发现缺陷** —— 禁止无效断言，每条断言都要能捕获被测服务的真实偏差。

---

## 1. 断言质量规则（分级 L0–L3）

### 等级定义

| 等级 | 特征 | 能否发现缺陷 | 通用示例 |
|------|------|:---:|------|
| **L0 禁止** | 仅存在性/非空 | 否 | `assert resp is not None` / `assert len(events) > 0` |
| **L1 最低** | 检查关键字段值 | 勉强 | `assert resp.json()["status"] == "OK"` |
| **L2 合格** | 结构 + 值语义 + 类型/数量 | 能 | 对字段取值/枚举/错误码做精确判定，且 id/数量/类型核对 |
| **L3 优秀** | L2 + 过程/状态/轨迹验证 | 高 | L2 + 流式末事件终态校验 + 复查接口一致 |

### 强制要求

- **所有断言 ≥ L1**：L0 不得作为最终断言。
- **正例（positive）用例 ≥ L2**：必须验证返回结构关键字段的**值语义**，而非仅存在性。
- **每个用例至少包含 1 条 L2（值语义）断言** —— 值语义=对字段取值/枚举/错误码做精确判定。
- **每个用例 ≥ 2 个维度**的断言：
  - **内容维**：返回值内容/结构/字段取值（如状态字段、`error.code`、发现端点字段）。
  - **过程维**：执行过程产生的事件序列/SSE 终态/状态前后对比/交互轨迹（如 SSE 末事件 = 终态、复查接口一致）。

> 断言所用的字段路径/枚举/错误码一律以 `contract.md` 为准（见 §2），通用容差访问器见 `reference/http_client.py`。

## 2. 判据来源纪律（最高红线）

**断言与期望值一律源自以下二者之一，二者之外一律视为臆测、禁止：**

| 合法来源 | 说明 |
|----------|------|
| **`contract.md`** | stage2.5 用真实 SUT 探活校准得到的响应契约（线缆形态、字段路径、枚举、错误码、SSE 形态、id 类型）。|
| **Grep 核实的源码** | 在被测 Java 仓中实际命中的注解/常量/枚举/序列化逻辑（标注 `文件:行号`）。|

### 禁止臆测响应形态

- 不得凭需求文档措辞或经验“猜”响应结构、字段嵌套、枚举写法、错误码、SSE 事件序列。
- 任何裸字段访问前，先确认 `contract.md` 记录的真实嵌套与命名；形态有容差时经访问器读取
  （通用 `id_eq`/`normalize_state`/`parse_sse_lines`，协议特化访问器见对应 `examples/`）。
- `contract.md` 未覆盖的断言点须标 `needs-runtime-verify`，不得据臆测放行。

> 教训：参考一旦写错，整批用例会继承同一批错误假设（garbage-in-garbage-out）。
> 所以本 skill 把“用真实 SUT 校准判据”（stage2.5）作为硬环节，而非信任手写参考。

## 3. 人工裁决补位规格库

某些判定需要**业务约束或架构知识**，仅靠静态扫描 + probe 无法确定（例如：某入口是否应拒绝、
某字段是否业务必填、某质量维度的边界阈值）。此类点必须**人工确认补位**，沉淀进**规格库**：

| 环节 | 强制人工确认 | 缺位规格库的原因 |
|------|--------------|------------------|
| **stage1 需求分析** | 是 | 缺业务约束知识：哪些是关键链路、哪些异常必须覆盖、验收口径 |
| **stage3b 用例设计** | 是 | 缺架构/契约判定知识：某断言点是 spec-required 还是观察项、边界取值 |

- 人工确认结论写入规格库（如 `.state/spec_decisions.json` 或文档），作为后续判据依据。
- **规格库就位后可降级**：同类判定一旦有规格库背书，后续可由 Agent 直接引用，不再强制逐次人工确认。

> 规格库是“判据来源纪律”的补充：当 contract.md/源码都无法回答“应该是什么”时，由人工裁决补位，并固化复用。

## 4. 执行边界五分类（不得用弱化断言“洗绿”）

每条执行结果必须归入且仅归入以下一类：

| 分类 | 触发条件 | 处理 |
|------|----------|------|
| **harness_defect** | 测试脚手架/用例代码自身缺陷（import 错、fixture 错、断言写错路径、客户端调用误用） | **自我修复**：修脚手架/用例后重跑，不计为 SUT 问题 |
| **sut_unsatisfied** | SUT 行为不满足某非协议必需的期望（功能未实现/可选能力缺失/业务约束未达） | **忽略 skip**：不计失败、不改断言；记录观察，不视为缺陷 |
| **sdk_defect** | SUT 违反协议必需约束 / 返回 5xx / 栈异常 / 契约违例（contract 确认形态应正确） | 标记为**确认缺陷**，记录证据进缺陷清单 |
| **env_issue** | 连接失败 / SUT 未就绪 / 依赖缺失（httpx、网络、服务未起） | 命中**就绪门**；标记，**保留代码**，不修改断言，不计失败 |
| **pass** | 断言全通过 | 计为通过 |

> **红线**：不得通过弱化由 contract 背书的断言（如把 L2 降成 L0、删去值语义判定）来让用例转绿。
> 转绿的唯一合法路径是“断言确实臆测错了，按 contract 校准”或“确属 harness_defect 自我修复”，
> 而非“判据本身不可信但强行放行”。`sut_unsatisfied` 与 `sdk_defect` 的区分**以 contract.md 是否将该约束列为 spec-required 为界**（见 §6）。

## 5. SUT 就绪门（stage4 前置）

stage4 生成/执行任何用例**之前**，编排器必须探活 SUT：

```
探活：GET <discovery/health 路径>（contract.md 指定，A2A 例为 /.well-known/agent-card.json）→ 2xx 且可解析
  ├── 就绪      → 进入 stage4 执行
  └── 不就绪    → 全部用例标记 env_issue，保留已生成代码，不计为失败 / 缺陷
```

> 目的：避免无服务时只得到 env_issue 却掩盖了形态假设错误。
> 就绪门把“环境不可达”（env_issue）与“判据不符”（sdk_defect/校准）彻底分开，二者绝不混为一谈。

## 6. spec-required vs deployment-config-dependent

字段断言强度按其规约地位分级（类别由 `contract.md` 标注）：

| 类别 | 定义 | 断言强度 | 通用示例 |
|------|------|----------|------|
| **spec-required** | 协议必需字段 | **硬断言**（违反即 `sdk_defect`） | 错误码、必填状态字段、id 回带 |
| **deployment-config-dependent** | 取值取决于部署配置 | **观察项**（记录，不作硬失败） | 依赖部署 URL/开关的字段，未配置时可空 |

> 部署配置相关字段为空通常是 `sut_unsatisfied`/观察项，而非缺陷；核心判据应落在 spec-required 字段。

---

## 7. 并发与隔离

| 规则 | 说明 |
|------|------|
| **1 用例 = 1 Agent** | 硬性约束，违反即任务失败回滚；即使用例来自同一设计文件也独立 Agent |
| **批次大小** | `--case-batch-size` 控制并发（默认 5），滑动窗口满载 |
| **Agent 间无状态** | 仅经文件协议传递；结果各自记录到 `case_results.json` |
| **轨迹一等公民** | 每条用例落 `trace/<case>.jsonl`，会话日志落 `trace/session.log`（设置 `A2A_TRACE_DIR` 启用）|

---

## 8. 禁止事项速查

| 禁止行为 | 正确做法 |
|----------|----------|
| 臆测响应形态 | 查 `contract.md` 或 Grep 源码核实 |
| 裸字段访问 SUT 响应（有容差时） | 用容差访问器（通用见 `reference/http_client.py`） |
| 弱化 contract 背书的断言以转绿 | 仅当断言确为臆测错误时按 contract 校准；harness 自身错则自我修复 |
| 仅 L0 断言 / 只有 1 条断言 | ≥L1 且 ≥2 维度，正例 ≥L2，含 ≥1 条值语义 |
| 1 个 Agent 处理多用例 | 1 用例 1 Agent |
| 无服务时仍判失败/缺陷 | 探活不就绪 → env_issue，保留代码 |
| 把部署配置相关字段当硬断言 | 按 deployment-config-dependent 记为观察项 |
| 缺业务/架构判据时凭感觉放行 | 走 §3 人工裁决补位规格库 |
