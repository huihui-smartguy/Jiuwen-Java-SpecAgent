# AutoTestFlow 演讲稿

## 一、开场

各位好。今天我向大家介绍 **AutoTestFlow——一个需求驱动的测试智能体**。

一句话概括它做的事：你给它一份需求文档（可选再附上被测代码和一个运行中的服务地址），它就自动地帮你把"分析需求、设计测试场景、设计测试用例、生成可执行的测试代码、真正跑起来、再出一份报告"这一整条链路跑完——并且生成的用例是**可信、可执行、可追溯**的。

## 二、为什么需要它：AI 时代测试的真问题

进入大模型时代，"造"测试用例的成本几乎降到了零：让模型写一堆看起来很像样的用例，几秒钟的事。但真正的难点变成了"**判对错**"——你怎么知道这些用例是对的、判据是准的？

我们在实践中反复踩到一个坑：只要模型对被测系统的**响应形态、错误码、状态机**理解有一点偏差，它生成的用例就会**成批失真**——一处参考写错，几十条用例跟着错（garbage in, garbage out）；更糟的是，在没有真实服务时，这些错误还会被"环境问题"掩盖，让人误以为是被测系统有缺陷。

所以 AutoTestFlow 的设计目标非常明确：**既要 AI 的高吞吐，又要判据的权威性**——让"造"用例交给 AI，让"判对错"锚定在真实的系统契约和人的裁决上。

## 三、它是什么：一条受约束的多阶段流水线

AutoTestFlow 不是让大模型自由发挥，而是一条**受约束的、可复现的多阶段流水线**，采用"**编排器 + 子 Agent + 文件即协议**"的架构：

- 主编排器只做调度，不读业务细节；
- 每个阶段由独立子 Agent 执行，彼此隔离；
- 所有数据通过文件传递，天然支持断点续跑。

它有**两种模式**：给了代码（或 PR/Commit）走"标准模式"，跑完整链路；只给需求、没有代码就走"纯需求模式"，到测试设计为止。

## 四、完整工作流程

标准模式下，它依次跑这几步：

1. **需求分析**：从需求文档提炼功能点与测试场景（流程、框架组合、质量三类），不读代码、避免被实现带偏。**完成后强制人工裁决**。
2. **代码扫描**：扫描 Java/Spring 代码，识别对外入口（控制器、注解、端点）、异常映射与错误码、状态枚举、响应序列化形态，并据此**派生框架场景**。
3. **契约校准**（关键的一步）：探活真实运行的被测服务、结合源码与 SDK，把响应的真实形态固化成一份权威契约 `contract.md`。
4. **场景补充与用例设计**：在需求场景之上补齐代码独有能力（GAP）和框架组合场景，再设计出结构化的测试用例；每条用例的判据都要能在 `contract.md` 找到依据。**完成后再次强制人工裁决**。
5. **生成与执行**：先做 P0 金标准用例、再批量生成，调用黑盒客户端真正打被测服务、执行、自动验证与修复，并记录交互轨迹。
6. **出报告**：覆盖设计覆盖度、执行结果分类、确认缺陷清单、被忽略项、以及交互轨迹摘要。

## 五、四个关键能力（这是它和"让模型随便写用例"的本质区别）

**第一，契约校准——判据权威性。** 在落任何断言之前，先用真实运行的服务把响应的真实形态、字段路径、枚举、错误码采下来，沉淀成权威契约；之后所有断言一律以这份**实测契约**为准，**绝不凭需求文档的措辞臆造响应结构**。这一条直接根治了"参考写错、用例全错"。

**第二，人工裁决补位规格库。** 在还没有"业务约束、架构知识"这类规格库的阶段，需求分析对不对、用例设计全不全，缺少客观判据。所以在**需求分析**和**用例设计**两个最关键的节点，设了**强制人工把关**，由人补位评判标准；将来规格库就位，这两道门可以降级为自动。

**第三，执行边界清晰——修复、忽略、还是缺陷。** 生成的用例不可能一次全过，关键是分清责任：
- 脚本**自身**出问题（导入、语法、接口用错）→ **自我修复**；
- 被测服务**当前确实不满足**某条用例（特性没实现、行为不同）→ **忽略并记录，不计为失败**；
- 被测真正报错或违反契约 → **上报为缺陷**；
- 服务没就绪、连不上 → 标为环境问题，保留代码、不误判。
  一句话：**脚本错就修，系统真不满足就忽略，绝不靠删改断言来"洗绿"。**

**第四，交互轨迹是一等公民。** 测试智能体与被测服务之间的每一次请求、响应、以及流式输出，都**逐帧实时打印并落盘**，全程可追溯，方便复盘与定位问题。

## 六、测试维度：现在与未来

当前它以**场景化测试**落地，覆盖流程驱动、框架组合、质量保障三类场景；同时在架构上为**DFX 测试**（性能、可靠性、安全、兼容性等）预留了**并行扩展**的轨道——也就是说，它不只是端到端，而是一个可以横向长出更多测试维度的测试智能体。

## 七、依赖内化：调用更简单

过去这类工具往往要先手工跑一堆"前置步骤"去生成依赖文件。AutoTestFlow 把这些都**内化**了：测试脚手架内置、判据契约自动校准、框架场景自动派生。所以你**不需要任何手工预生成**，调用时只要一条命令：

```
/auto-test-flow 需求.md <被测仓>/<模块> --sut-base-url http://host:port
```

剩下的设计、生成、执行、报告，它自己跑完。

## 八、它是被真实验证过的

这套思路不是纸上谈兵。我们用一个真实的 Java 服务（A2A 协议特性）做过端到端验证：第一轮，模型按"想当然"的响应形态生成的用例**全军覆没**；定位后发现问题全在测试侧的形态假设，而不是被测系统——这正是"契约校准 + 人工裁决 + 交互轨迹"这几个机制要解决的。补齐之后再跑，用例**全部通过**，并且每一次交互都有迹可循。AutoTestFlow 就是把这套来之不易的经验，固化成了一个可复用的技能。

## 九、收束

最后用一句话总结 AutoTestFlow：**把机械的"造用例"交给 AI 去高吞吐，把"判对错"交给真实契约和人的裁决，让生成的测试可信、可执行、可追溯。** 谢谢大家。
